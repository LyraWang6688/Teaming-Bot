## 1. 系统概述

本系统基于 **Amy Edmondson Teaming 框架** 与 **Bion 群体动力学**，对会议文字稿进行结构化分析。当前项目中，网页上传任务和飞书会议自动分析都复用同一套分析内核：

- 统一入口：`src/services/analysisService.ts`
- 系统提示词：`src/constants/teamingRules.ts`
- Zone 决策树与一致性规则：`src/constants/zoneConfig.ts`
- 输出结构：`src/types/index.ts`

### 当前核心模型

| 项目 | 当前实现 |
|------|---------|
| LLM 模型 | `doubao-seed-2-0-pro-260215` |
| 调用方式 | 最多 3 次重试 |
| 输出格式 | 强约束 JSON Schema |
| 分析目标 | 不只解释“说了什么”，而是解释“团队此刻处于什么动力状态” |

### 核心设计原则

- Zone 由行为决策树判定，不由 PS/WS 分数直接决定。
- PS/WS 分数仅作为解读透镜和前端图表展示坐标。
- 分析结果必须同时服务于网页上传链路和飞书自动化链路，因此分析服务不绑定具体输入来源。

## 2. 分析流程总览

```text
会议文字稿
  │
  ├─ 入口1：网页上传异步任务
  │   ├─ POST /api/analyze/tasks 创建 web_analysis_tasks
  │   ├─ 后台任务解析文件 / 提取文字
  │   └─ 调用 analyzeMeetingText()
  │
  ├─ 入口2：飞书会议流水线
  │   ├─ 妙记转写导出
  │   └─ 调用 analyzeMeetingText()
  │
  └─ analyzeMeetingText()
      ├─ 发送系统提示词 + JSON Schema 给 LLM
      ├─ 解析/修复 LLM JSON 返回
      ├─ 校验 behaviors / teamState 是否存在
      ├─ 根据项目阶段 + qualityFlag 做 Zone 决策树修正
      ├─ 必要时重写 teamState.analysis
      ├─ 重算 communication.speakingShare
      ├─ 同步更新 dialogueNetwork.nodes.speakingShare
      ├─ 输出行为一致性告警日志
      └─ 返回 AnalysisResult
```

### 输出结果的关键新增项

相较旧版结构，当前分析结果新增：

- `dialogueNetwork`
- `unfinishedDialogues`
- `unseenDisagreements`
- `keyAssumptions`

它们共同支撑新版报告模板的四部分结构。

## 3. LLM系统提示词与输出结构

### 3.1 提示词文件位置

`src/constants/teamingRules.ts`

### 3.2 提示词目标

系统提示词要求 LLM 同时完成以下任务：

- 判断会议类型、项目阶段、有效发言数与 `qualityFlag`
- 输出四个行为维度的 level / score / evidence / summary
- 输出 `teamState` 及三段式 `analysis`
- 输出领导者建议
- 输出互动网络
- 输出未完形对话、非共识、关键假设

### 3.3 当前 JSON Schema 约束重点

1. `summary`
   - 只总结“会议讨论了什么”，不直接写团队动力结论。
2. `teamState.analysis`
   - 必须按 `【区域判断】`、`【工作标准】`、`【心理安全感】` 三段输出。
3. `leaderAdvice.advice`
   - 必须给一个具体动作，并包含可直接对团队说出的台词。
4. `unseenDisagreements.whyItMatters`
   - 只能讲“为什么值得被看见”，禁止转成风险预警或执行建议。
5. `dialogueNetwork.nodes.playerRole`
   - 当前类型系统接受 `mover | follower | opposer | bystander | silent`，前端以此渲染网络角色。

### 3.4 返回解析策略

LLM 可能返回以下异常格式：

- 包裹在 Markdown code block 中
- JSON 字符串内部换行未转义
- 尾部多出说明文字
- JSON 被截断

`analysisService.ts` 通过以下流程兜底：

- 提取 code block 内正文
- 截取首个 `{` 与最后一个 `}`
- `repairJsonStrings()` 修复字符串中的换行和未转义双引号
- `generateTruncationFixes()` 尝试补全缺失的 `]` / `}`
- 仍失败时记录 `llm_response_parse_failed`

## 4. 核心概念：Zone四象限模型

### 4.1 四象限定义

```text
                    高工作标准
                        │
                        │
          Anxiety       │      Learning
          (焦虑区)       │      (学习区)
          (低安全        │      (高安全
           +高标准)      │       +高标准)
     ────────────────────┼─────────────────────→ 高心理安全感
          Apathy        │      Comfort
          (冷漠区)       │      (舒适区)
          (低安全        │      (高安全
           +低标准)      │       +低标准)
                        │
                        ↓
                    低工作标准
```

### 4.2 决策原则

当前项目明确采用“行为色标 -> 决策树 -> 最终 Zone”的方法：

```text
S(直言不讳) = H/M ?
├─ 是 → E(实验) ≥ M 且 R(反思) ≥ M ?
│       ├─ 是 → Learning
│       └─ 否 → Comfort
└─ 否 → E ≥ M 或 R ≥ M ?
        ├─ 是 → Anxiety
        └─ 否 → Apathy
```

色标映射规则：

- `Green -> H`
- `Blue -> M`
- `Red -> L`
- `Grey -> L`

说明：

- `collaboration` 不参与决策树分叉，但影响整体叙述质地和相关得分。
- `psychologicalSafetyScore` / `workStandardScore` 不用于 Zone 判定。

### 4.3 强制返回 Difficult to Judge 的情况

满足以下任一条件时，直接返回 `Difficult to Judge`：

- `effectiveSentences < 10`
- `determineZoneFromBehaviors()` 判定结果为难以判断

此外，`qualityFlag = low_sample` 时，会阻止低样本会议被轻易判为学习区。

## 5. 行为评估（Behaviors）

### 5.1 四个行为维度

| 维度 | 中文名 | 观察重点 |
|------|-------|---------|
| `speakingUp` | 直言不讳 | 是否敢表达真实问题、担忧、不同意见 |
| `collaboration` | 协同 | 是否互相配合、对齐目标、连接上下文 |
| `experimentation` | 实验 | 是否承认不确定性并用试探性行动推进 |
| `reflection` | 反思 | 是否对过程、结果、做法进行批判性检视 |

### 5.2 Level 语义

| Level | 语义 |
|-------|------|
| `Red` | 明显不足或存在风险 |
| `Blue` | 基本到位，但还不够深 |
| `Green` | 明显健康且持续出现 |
| `Grey` | 证据不足，不宜判断 |

### 5.3 一致性约束

行为 `summary` 必须与 `level` 保持一致：

- `Red` 不能写成积极赞扬
- `Blue` 不能写成卓越成熟
- `Green` 不能写成明显缺失
- `Grey` 必须承认数据不足

## 6. Zone修正与一致性验证

### 6.1 修正顺序

`analyzeMeetingText()` 中的 Zone 处理顺序如下：

1. 接收 LLM 原始返回的 `teamState.zone`
2. 计算 `inferZoneFromScores()` 并记录“分数推导 vs LLM 结果”差异日志
3. 再调用 `determineZoneFromBehaviors()` 得到最终 Zone
4. 若和原始 Zone 不一致，以行为决策树结果为准
5. 输出 `zone_overridden_by_behavior` 日志

### 6.2 一致性检查

在最终 Zone 确定后，系统会逐个校验四项行为是否和该 Zone 的期望相匹配：

```typescript
const validation = validateBehaviorLevel(zone, key, level);
```

如果不一致：

- 只记录告警，不强制改值
- 告警事件为 `behavior_level_zone_mismatch`

设计原因：

- 最终 Zone 必须稳定、可解释
- 避免“为了修一致性而反复改动行为字段”导致循环修正

### 6.3 日志输出

分析服务会记录一份行为一致性摘要，例如：

- 团队基调说明
- 每项行为是否一致
- 不一致时的期望 level 和当前 level

## 7. 分析重写与容错解析

### 7.1 为什么需要重写 `teamState.analysis`

LLM 返回的 `teamState.analysis` 可能仍按原始 Zone 的语气书写。例如：

- LLM 初判 `Learning`
- 决策树修正为 `Comfort`

此时如果不重写，报告文本会和最终 Zone 自相矛盾。

### 7.2 重写策略

优先级如下：

1. `rewriteAnalysisWithLLM()`
   - 使用修正后的 Zone、行为摘要、PS/WS 分数与会议摘录重新生成文字
   - 温度为 `0.7`
   - 明确禁止暴露分数和审计逻辑
2. `regenerateAnalysis()`
   - 当重写调用失败时，使用模板式兜底文本

### 7.3 三段式书写约束

新版 `teamState.analysis` 必须能支撑前端拆出三段：

- `【区域判断】`
- `【工作标准】`
- `【心理安全感】`

前端会优先按标签拆分，标签缺失时再尝试按自然语言锚点兜底。

## 8. 发言占比与互动网络后处理

### 8.1 `recalculateSpeakingShare()`

LLM 给出的发言占比不直接信任，服务端会二次计算：

1. 优先识别转写文本中的说话人时间戳
2. 按相邻发言时间差估算持续时长
3. 最后一个发言者用中位数时长补尾
4. 若时间戳不可靠，则回退到按字符数估算

### 8.2 名称匹配策略

为把转写文本中的说话人与结构化输出中的参与者对齐，系统会进行：

- 去空格
- 小写归一化
- 去掉 `@`
- 处理“说话人X用某某的设备”一类名称
- 包含关系匹配

### 8.3 与 `dialogueNetwork` 的联动

当 `communication.speakingShare` 计算完成后，会同步回写到：

```typescript
dialogueNetwork.nodes[i].speakingShare
```

这样网络图上的百分比和参与者列表使用同一份后处理结果。

## 9. 给领导者的建议（LeaderAdvice）

### 9.1 输出结构

```typescript
leaderAdvice: {
  action: 'frame_for_learning' | 'create_psychological_safety' | 'learn_from_failure' | 'cross_boundaries',
  advice: string,
  reasoning: string
}
```

### 9.2 当前要求

- `action` 必须是四项行动之一
- `advice` 必须足够具体，最好包含可直接说出口的话
- `reasoning` 必须说明“为什么这项行动最优先”

### 9.3 验证方式

当前服务端只做基础完整性校验：

- `action` 是否存在
- `advice` 是否存在
- `reasoning` 是否存在

若不完整，会记录 `leader_advice_incomplete`

## 10. 项目阶段对分析的影响

### 10.1 阶段字段

`metadata.projectPhase` 取值：

- `Start-up`
- `Post-startup`

### 10.2 启动期影响

在启动期：

- `speakingUp` 和 `collaboration` 的判断标准相对宽松
- `experimentation` 与 `reflection` 更容易被标记为 `Grey`

但要注意：

- `Grey` 在决策树中依然按 `L` 处理
- 启动期不会因为“天然缺实验/反思”而被特殊豁免进学习区

这保证了“缺证据 != 自动给中等表现”。

## 11. 后端入口与任务链路

### 11.1 统一分析入口

- `src/services/analysisService.ts`：统一分析内核

### 11.2 网页上传链路

- `src/app/api/analyze/tasks/route.ts`：创建异步任务
- `src/app/api/analyze/tasks/[taskId]/route.ts`：轮询状态
- `src/lib/analysis/inputParser.ts`：解析 `.txt` / `.docx`
- `src/lib/analysis/webAnalysisTaskProcessor.ts`：后台执行分析

### 11.3 飞书会议链路

飞书链路不修改分析内核，只负责把转写文字交给 `analyzeMeetingText()`，因此自动化外壳与分析引擎保持解耦。

### 11.4 `web_analysis_tasks`

该表用于承接网页上传异步分析任务，避免浏览器请求长时间挂起。关键字段包括：

- `id`
- `status`
- `meeting_text`
- `result`
- `attempt_count`
- `last_error_type`
- `last_error_message`
- `started_at`
- `completed_at`
