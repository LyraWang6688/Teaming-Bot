## 1. 系统概述

本系统是一个基于 **Amy Edmondson Teaming 框架** 的会议纪要AI分析工具，结合 **Bion群体动力学理论** 进行深度诊断。

### 核心技术栈
- **LLM模型**：豆包大语言模型（doubao-seed-1-8-251228）
- **Temperature**：1（高多样性）
- **框架**：Amy Edmondson的心理安全感和工作标准四象限模型

### 支持的文件格式
| 格式 | 解析方式 | 说明 |
|------|---------|------|
| `.txt` | 原生文本读取 | 纯文本会议纪要 |
| `.docx` | mammoth 库解析 | Word 文档，提取纯文本内容 |

### 文件大小限制
- 单文件最大 **10MB**

### 核心目标
> 你的任务不是"统计行为"，而是"诊断团队的生存状态"。
> 关注互动的**质量**、发言的**主动性**、共识的**真实性**。

## 2. 分析流程总览

```
文件上传（.txt / .docx）
    │
    ├─────────────────────────────────────────────────────────────┐
    │ 第零阶段：文件解析与验证                                      │

    │   - 文件大小校验（≤10MB）                                     │
    │   - .docx 使用 mammoth 提取纯文本                              │
    │   - .txt 直接读取文本                                          │
    └─────────────────────────────────────────────────────────────┘
    │
    ├─────────────────────────────────────────────────────────────┐
    │ 第一阶段：文本预处理与有效性判定                              │
    │   - 清洗无效发言（语气词、流程性短语）                          │
    │   - 有效发言判定（≥10字符，非流程性）                          │
    │   - 质量标记（normal/unbalanced/low_sample）                  │

    └─────────────────────────────────────────────────────────────┘
    │
    ├─────────────────────────────────────────────────────────────┐
    │ 第二阶段：项目阶段判断（必须首先执行）                          │
    │   - 判断 "Start-up" 或 "Post-startup"                       │
    │   - 影响后续 Behavior 的期望标准和 Learning Zone 判定          │
    └─────────────────────────────────────────────────────────────┘
    │
    ├─────────────────────────────────────────────────────────────┐
    │ 第三阶段：快速排除诊断（必须优先执行）                          │
    │   - 直言不讳率检查（结合工作标准综合判断）                      │
    │   - 共识真实性检查（伪共识 → 排除舒适区/学习区）                 │
    │   - 互动深度检查（无深度互动 → 排除学习区）                     │

    │
    ├─────────────────────────────────────────────────────────────┐
    │ 第四阶段：证据权重评估                                        │

    │
    ├─────────────────────────────────────────────────────────────┐
    │ 第五阶段：心理安全感严苛审计（X轴）                            │

    │   - 分数 = 10 - 信号数×1.5（仅辅助展示）                          │

    │
    ├─────────────────────────────────────────────────────────────┐
    │ 第六阶段：工作标准严苛审计（Y轴）                              │
    │   - 7项低工作标准红线信号                                     │
    │   - 基准分10分，每项扣1.5分                                   │
    │   - 分数 = 10 - 信号数×1.5（仅辅助展示）                          │
    └─────────────────────────────────────────────────────────────┘
    │
    ├─────────────────────────────────────────────────────────────┐
    │ 第七阶段：模式验证（最高优先级）                               │
    │   - 对照各Zone的模式签名进行验证                              │
    │   - 验证失败 → 禁止判断为该Zone                               │
    └─────────────────────────────────────────────────────────────┘
    │
    ├─────────────────────────────────────────────────────────────┐
    │ 第八阶段：行为级别判定                                        │

    └─────────────────────────────────────────────────────────────┘
    │
    └─────────────────────────────────────────────────────────────┐
    │ 后端验证（两步走）                                             │
    │   步骤1：以决策树确定最终 Zone（Behavior色标→决策树）            │
    │   步骤2：验证一致性（仅警告，不修正）                            │
    │   + leaderAdvice 基于四项领导力行动生成建议                    │
    └─────────────────────────────────────────────────────────────┘
```

## 4. 核心概念：Zone四象限模型

### 四象限定义

```
                    高工作标准
                        │
                        │
          Anxiety       │      Learning
          (焦虑区)       │      (学习区)
          (低安全        │      (高安全
           +高标准)      │       +高标准)
     ────────────────────┼───────────────────── 低工作标准
          Apathy        │      Comfort
          (冷漠区)       │      (舒适区)
          (低安全        │      (高安全
           +低标准)      │       +低标准)
                        │
                        │
                        │
                    高心理安全感
```

### Zone特征总结

| Zone | 心理安全感 | 工作标准 | 核心特征 | 典型Behavior |
|------|-----------|---------|---------|-------------|
| **Apathy（冷漠）** | 低 | 低 | "用最小努力完成工作"，观望态度 | 全部Red/Grey |
| **Comfort（舒适）** | 高/中 | 低/中 | 安全但没标准，"做自己舒服但没理由寻求额外挑战" | SpeakUp/Collab: Green/Blue<br>Exper/Reflec: Red |
| **Anxiety（焦虑）** | 低/中 | 高/中 | 管理者将高标准等同于好的管理，员工不敢提出想法、不敢尝试新的程序、不敢寻求帮助 | SpeakUp: Red<br>其他: Blue/Red（被动/交差式） |
| **Learning（学习）** | 高 | 高/中 | 容易协同配合、相互学习、完成工作 | 全部Green/Blue |

> **重要**：Edmondson 原著明确指出，心理安全感(PS)和责任度/工作标准(WS)是两个独立维度，不是连续光谱的两极。

### Zone判断：决策树法

**核心原则**：Zone 的判定完全由决策树从行为色标推导，PS/WS 分数不参与 Zone 判定，仅用于 analysis 文字的书写。

**决策树逻辑**：

```
S(直言不讳) = H 或 M？
├─ 是 → E(实验) ≥ M 且 R(反思) ≥ M？
│       ├─ 是 → 学习区（Learning）
│       └─ 否 → 舒适区（Comfort）
└─ 否 → E(实验) ≥ M 或 R(反思) ≥ M？
        ├─ 是 → 焦虑区（Anxiety）
        └─ 否 → 冷漠区（Apathy）
```

**色标到等级的映射**：Green→H, Blue→M, Red→L, Grey→L（统一视为L，包括启动期）

**决策树解读**：
- **第一层分叉看直言不讳**：S 是心理安全感的核心指示器——敢不敢说是安全感最直接的体现
- **第二层分叉看实验和反思**：工作标准的核心是学习闭环（实验→反思），两者缺一不可
- **舒适区 vs 学习区**：同样是"敢说"（S≥M），区别在于是否往深走（E≥M且R≥M）
- **焦虑区 vs 冷漠区**：同样是"不敢说"（S=L），区别在于是否有工作标准在驱动（E≥M或R≥M）
- **协同(C)不参与决策树分叉**：但影响报告文字质地描述和分数计算

**81种排列分布**：
- 学习区：24种（S=H/M × E≥M × R≥M）
- 舒适区：30种（S=H/M × E和R不都≥M）
- 焦虑区：24种（S=L × E≥M或R≥M）
- 冷漠区：3种（S=L × E=L × R=L）
- 合计：24+30+24+3=81 ✓

**PS/WS 分数的角色**：
- ❌ 不参与 Zone 判定（Zone 由决策树从行为色标判定）
- ✅ 用于 analysis 文字书写：PS/WS 为解读视角（非格式化模板），行为评估在需要时引入佐证
- ✅ 用于前端展示：四象限图中显示分数对应的位置

## 5. 行为评估（Behaviors）

### 四个行为维度

| 维度 | 中文名 | 核心问题 |
|-----|-------|---------|
| **speakingUp** | 直言不讳 | 个体之间是否有坦诚、直接的对话？是否提出问题、寻求反馈、讨论错误和担忧？ |
| **collaboration** | 协同 | 团队是否拿出协同的心态和行为？是否包含内外部的信息分享与行动协调？ |
| **experimentation** | 实验 | 团队是否以试探性、迭代的方式行动？是否承认不确定性并从行动结果中学习？ |
| **reflection** | 反思 | 团队是否批判性地检视行动结果？是否持续观察、提问与讨论流程和产出？ |

### Level语义定义

| Level | 含义 | 典型特征 |
|-------|-----|---------|
| **Red（警惕）** | 不足 | 缺乏坦诚对话、协同流于表面、回避不确定性、缺少批判性检视 |
| **Blue（良好）** | 基本到位 | 有基本沟通与配合、面对不确定时偶有试探、偶有反思但不够系统 |
| **Green（健康）** | 充分 | 坦诚且深入的对话、主动跨界协同、拥抱不确定并从中学习、持续批判性检视 |
| **Grey（难以判断）** | 数据不足 | 发言数量不足 |

### Level与Summary一致性约束

```
⚠️ 严禁出现 level 与 summary 语义不一致：

Red → summary必须描述不足          ❌ "团队表现良好"
Blue → summary描述基本正常          ❌ "团队表现卓越"
Green → summary描述卓越表现         ❌ "团队缺乏主动性"
Grey → summary说明数据不足          ❌ "团队积极发言"
```

## 6. 快速排除诊断机制

### 6.1 直言不讳率检查

```
直言不讳率 = 坦诚发言（提出问题、寻求反馈、讨论错误和担忧）/ 总发言数
坦诚发言 = 自发提出观点、承认困难、质疑决策、分享担忧（非被点名、非场面话）

判定规则（结合工作标准综合判断）：
- 直言不讳率 < 30% + 工作标准低 → 强烈指向"冷漠区"
- 直言不讳率 < 30% + 工作标准较高 → 指向"焦虑区"（高压下的被动执行）
- 直言不讳率 < 30% + 氛围轻松 → 指向"舒适区"（缺乏挑战但愿意参与）
- 注意：直言不讳率低本身不足以单独决定Zone，需结合工作标准和互动氛围
```

### 6.2 共识真实性检查

```
检测伪共识信号：
- 成员表达困难（忙碌、资源不足）
- 但仍被要求接受任务
- 未讨论可行性或调整方案

判定规则：
- 检测到伪共识 → 排除"学习区"和"舒适区"，在焦虑区或冷漠区中判断
- 伪共识 + 高工作标准 → 指向焦虑区
- 伪共识 + 低工作标准 → 指向冷漠区
- 注意：焦虑区也可能出现伪共识，不应因此排除焦虑区
```

### 6.3 互动深度检查

```
深度互动信号：
- 主动提问（质疑性、探索性）
- 主动挑战他人观点
- 超出事务层面的探索性对话

判定规则：
- 无深度互动 → 直接排除"学习区"
- 仅事务性讨论 → 强烈指向"舒适区"
- 学习区必须有至少1次深度互动
```

### 6.4 互动模式检查

```
对话结构分析：

星形结构（轮辐式）：
- 80%以上对话经过Leader
- 缺乏成员横向互动
- 若Leader停止，讨论即中断

网状结构：
- 成员间存在横向对话链
- 成员主动回应其他成员
- 无需Leader持续驱动

判定规则：
- 星形结构 + 被动回应 → 排除"学习区"
```

### 6.5 外部驱动检测

```
外部角色：助教、教练、顾问、客户、专家

区分"主导"与"引导"：
- 外部主导：外部角色直接给方案、下指令、做决策，团队仅被动执行 → 扣工作标准分数2-3分
- 外部引导：外部角色提出问题后，团队成员之间有多轮互相讨论（网状互动） → 不扣分，说明有引导思考的习惯
- 判断特征：提问之后，团队成员之间是否有多轮互相讨论（而非只对外部角色回应）

判定规则：
- 外部主导 + 星形结构 + 团队完全被动 → 强烈指向"焦虑区"
- 外部引导 + 团队主动网状讨论 → 可能是学习区或舒适区
```

### 6.6 发言均衡性检查

```
均衡定义：
- 单一发言者占比 ≤ 60%
- 至少3个成员有实质性发言

判定规则：
- 单一发言者 > 60% → 直接禁止判断为"学习区"
- 理由：学习区需要多元化参与
```

## 7. 分数计算与 Zone 判定

### 7.1 心理安全感审计（X轴）

**7项低心理安全感红线信号**：
1. 提问少，仅被动接受或汇报
2. 回避困难对话和敏感话题
3. 领导主导，缺乏权力下放
4. 缺乏建设性反馈
5. 不互相求助，存在自我防御
6. 几乎没有分歧（伪共识）
7. 仅限工作关系，缺乏心理连接

**计算规则**：
```
基准分：10分
每项信号：扣1.5分

最终分数计算公式：
  分数 = 10 - 信号数 × 1.5

示例：
  2项信号 → 10-3=7分
  3项信号 → 5.5分
  4项信号 → 4分

⚠️ 注意：分数仅用于 analysis 文字书写和前端展示，不参与 Zone 判定。
Zone 100% 由决策树决定（见第4节）。
已废除"≥3项信号→分数≤4分"的强制限制，避免分数与行为分析冲突。
```

### 7.2 工作标准审计（Y轴）

**7项低工作标准红线信号**：
1. 结论模糊（"尽快"、"优化"），无量化指标
2. 主观决策（"我觉得"、"以前就这样"）
3. 责任归属缺失（"我们"、"有人"）
4. 归咎于外/归咎于人
5. 回避深度挑战
6. 问题循环重复，无改进闭环
7. 容忍低效协作

**计算规则**：
```
基准分：10分
每项信号：扣1.5分

最终分数计算公式：
  分数 = 10 - 信号数 × 1.5

示例：
  2项信号 → 10-3=7分
  3项信号 → 5.5分
  4项信号 → 4分

⚠️ 注意：分数仅用于 analysis 文字书写和前端展示，不参与 Zone 判定。
Zone 100% 由决策树决定（见第4节）。
已废除"≥3项信号→分数≤4分"的强制限制，避免分数与行为分析冲突。
```

### 7.3 Zone判定逻辑

```
Zone 完全由决策树从行为色标推导，PS/WS 分数不参与判定。

决策树（完整版）：
  S=H/M?
  ├─ 是 → E≥M 且 R≥M? → 学习区
  │       └─ 否 → 舒适区
  └─ 否 → E≥M 或 R≥M? → 焦虑区
          └─ 否 → 冷漠区

硬性阈值（优先于决策树）：
  - Grey≥2 → Difficult to Judge
  - effectiveSentences<10 → Difficult to Judge

特殊规则（决策树之后）：
  - qualityFlag=low_sample → 学习区降级为舒适区
```

### 7.4 分数在 analysis 中的使用

```
分数是 analysis 叙事的主线，行为评估作为佐证。

叙事结构：
1. PS/WS 为解读视角（非格式化模板）
   - 不要机械地"先讲PS再讲WS"，而是从会议中最突出、最有洞察的现象切入，让 PS/WS 在叙述中自然浮现
   - 每场会议有自己的气质，analysis 应捕捉这种独特气质而非套用固定句式
   - 分数在行文中自然融入（如"心理安全感仅3分"），但不作为开场白或固定锚点
2. 行为评估在需要时引入佐证
   - 当 PS/WS 的判断需要佐证时，引入四项行为的具体表现作为支撑
   - 如果叙述本身已经清晰，不必强行凑齐四个维度
   - 例："不敢提出想法、不敢寻求帮助——对话受限"佐证 PS 不足
   - 例："成员间有坦诚对话，也能以协同心态配合"佐证 PS 尚可
3. 当分数指向与行为判定的 Zone 不一致时，以行为判定为准，但可在 analysis 中点出落差
```

## 8. 一致性验证机制

### 8.1 Zone判定优先级（两步走）

```
核心原则：Zone 由决策树从行为色标推导，分数是 analysis 解读视角（PS/WS 为透镜，行为为佐证）

执行顺序：
  步骤1：用决策树从行为色标推导 Zone → 最终 Zone
  步骤2：验证 Behavior 与 Zone 一致性（仅警告，不修正）
```

### 8.2 步骤1：决策树确定最终 Zone + 同步重写analysis

```typescript
// 后端验证逻辑（route.ts）
// 以决策树确定最终Zone，传入项目阶段信息和质量标记
const isStartup = parsed.metadata?.projectPhase === 'Start-up';
const qualityFlag = parsed.metadata?.qualityFlag;

// 硬性阈值：有效发言过少时强制返回 Difficult to Judge
const effectiveSentences = parsed.metadata?.effectiveSentences || 0;
if (effectiveSentences > 0 && effectiveSentences < 10) {
  parsed.teamState.zone = 'Difficult to Judge';
  parsed.teamState.analysis = regenerateAnalysis(
    'Difficult to Judge', parsed.teamState?.zone || 'Difficult to Judge',
    parsed.behaviors, parsed.teamState.psychologicalSafetyScore, parsed.teamState.workStandardScore
  );
} else {
  const zoneFromBehavior = determineZoneFromBehaviors(parsed.behaviors, isStartup, qualityFlag);

  if (zoneFromBehavior !== 'Difficult to Judge') {
    const originalZone = parsed.teamState?.zone;
    parsed.teamState.zone = zoneFromBehavior;  // 最终Zone

    // 如果 Zone 被修正，重新生成 analysis 使其与最终 Zone 一致
    if (zoneFromBehavior !== originalZone) {
      const newAnalysis = regenerateAnalysis(
        zoneFromBehavior,
        originalZone,
        parsed.behaviors,
        parsed.teamState.psychologicalSafetyScore,
        parsed.teamState.workStandardScore
      );
      parsed.teamState.analysis = newAnalysis;
    }
}
```

**determineZoneFromBehaviors 内部逻辑（决策树法）**：

1. **行为色标 → 维度等级**：Green→H, Blue→M, Red→L；Grey→L（统一视为L，包括启动期）
2. **决策树推导 Zone**：
   - S=H/M? → 是：E≥M且R≥M?→学习区，否→舒适区
   - S=H/M? → 否：E≥M或R≥M?→焦虑区，否→冷漠区
3. **硬性阈值**：Grey≥2 → Difficult to Judge; effectiveSentences<10 → Difficult to Judge
4. **特殊规则**：qualityFlag=low_sample → 学习区降级为舒适区

**设计原则**：
- Zone 是基于行为色标的**直接推导**，不再经过 PS/WS 中间层
- 决策树覆盖 81 种行为排列组合，保证 MECE（互斥且完备）
- PS/WS 分数是 analysis 解读视角（PS/WS 为透镜，行为为佐证），不参与 Zone 判定
- 协同(C)不参与决策树分叉，但影响报告文字质地描述和分数计算

**analysis同步重写机制**：

当后端修正Zone后，LLM原始生成的 `teamState.analysis` 可能仍基于旧Zone的措辞（例如Zone从"学习区"修正为"舒适区"，但analysis仍写着"团队积极学习"）。为此，后端会在Zone修正时**自动重新生成analysis**。

重写逻辑（优先级递减）：

1. **LLM 重写（首选）**：调用 `rewriteAnalysisWithLLM` 函数
   - 将修正后的 Zone + 四项行为色标 + PS/WS 分数 + 会议纪要摘要传给 LLM
   - LLM 基于这些信息重写 analysis，文字自然流畅
   - 使用较低 temperature (0.7) 确保输出稳定
   - 书写要求：PS/WS 为解读视角（非格式化模板），从会议最突出的现象切入，行为评估在需要时引入佐证，语气专业自然像咨询师说话
   - **绝对禁止**出现分数计算、审计逻辑、判定规则等字眼

2. **硬模板兜底（fallback）**：当 LLM 调用失败时，使用 `regenerateAnalysis` 函数
   - 不调用LLM，使用模板拼接生成，不增加耗时
   - 基于最终Zone生成自然叙述（每个Zone有独特气质的开篇：冷漠区从"低活跃"切入、舒适区从"表面和谐"切入、焦虑区从"紧张感"切入、学习区从"良性循环"切入）
   - 附上四个行为维度的level和summary（复用LLM已有的分析结果）
   - 如果Zone未被修正，analysis保持LLM原文不变

**analysis同步重写时分数的使用**：
- 重写 analysis 时，PS/WS 分数作为解读视角传入 LLM
- LLM 以分数为主轴展开论述（如"心理安全感仅3分，工作标准7分"），行为评估作为佐证
- 可对比"分数指向"与"行为判定"的差异，为读者提供更丰富的信息层次

### 8.3 步骤2：一致性检查（仅警告）

#### 8.3.1 Zone与行为的期望级别映射

> **注意**：此表仅用于一致性检查（步骤2）中的参考比对，不再用于 Zone 判定。Zone 判定由决策树完成（见第4节）。

`ZONE_BEHAVIOR_EXPECTATIONS` 定义了每个Zone下各行为维度的期望色标，一致性检查以此为准：

| Zone | 直言不讳 | 协同 | 实验 | 反思 | 依据 |
|------|---------|------|------|------|------|
| **Learning** | 绿/蓝 | 绿/蓝 | 绿/蓝 | 绿/蓝 | S≥M+E≥M+R≥M：安全且有标准，坦诚+协同+试+反思 |
| **Comfort** | 绿/蓝 | 绿/蓝 | 红 | 红 | S≥M但E和R不都≥M：敢说能配合但"没理由寻求额外挑战"，所以不试不反思 |
| **Anxiety** | 红 | 蓝/红 | 红 | 蓝/红 | S=L但E或R≥M：不敢提出想法、不敢尝试新程序、不敢寻求帮助；配合和反思是被动的、交差式的 |
| **Apathy** | 红/灰 | 红/灰 | 红/灰 | 红/灰 | S=L且E=L且R=L：观望态度，用最小努力完成工作 |

- 焦虑区 vs 学习区：看直言不讳——焦虑区不敢说真话（S=L），学习区敢说（S≥M）

## 9. 给领导者的建议（LeaderAdvice）

### 9.1 理论基础

基于 Amy Edmondson「组织学习（Organize to Learn）」框架的四项领导力行动。这四项行动不是孤立的工具，而是一套组合拳——组队和学习不会自然而然发生，领导力是帮助团体实现共同理解和协同行动的支点。

#### 行动1：为学习构建框架（Frame for Learning）

**核心要义**：框架即诠释——人们依靠它来感知与理解自身所处的环境。大多数情况下，框架是自动发生的，且默认是**自我保护的**，极大抑制协作、学习和改进的机会。领导者的力量在于重构框架，从自我保护转向以学习为导向。

**三个维度同时作用**：
- **领导者的角色**：明确表达互赖关系，承认自己也会出错，需要与大家一起协作
- **成员的角色**：被精挑细选的、对项目成功至关重要的专业人才
- **项目的意义**：既要有说服力、兼具挑战性，又不能太有挑战以致引起群体怀疑或无助

**可操作步骤**：招募→准备→试跑→反思（四步迭代）
**强化策略**：口头解释+视觉展示；接地气语言解释期待行为；发起促进新常规的活动；使用人工饰物视觉强化

#### 行动2：营造心理安全感（Create Psychological Safety）

**核心要义**：以人际信任和相互尊重为底色的工作氛围。关键不是"敢不敢说话"，而是**对人际关系风险后果的感知**——当你的发言可能让你被视为无知、无能、消极或有破坏性时，他人将作何反应？

**重要澄清**：PS不是做"老好人"，不是降低绩效标准。高度的心理安全感和高度的责任度可以并存。PS意味着——只要犯的错、提的问题是为了实现有野心的绩效目标，就不会受到惩罚或羞辱。

**七大好处**：鼓励直言→清晰思维→建设性冲突→缓冲失败→促进创新→消除高绩效障碍→增强责任感

**领导者具体行动**：贴近员工、平易近人；承认知识局限、展示可错面；邀请参与；把失败当学习机会；说话不绕弯；设边界并对越界行为问责

#### 行动3：学会从失败中学习（Learn from Failure）

**核心要义**：失败是组队和组织学习的必要方面。广义的失败包括大大小小不按计划进行的事件。逻辑上理解失败不可避免，但情感上很难把失败和责备剥离开——这导致惩罚性反应，进而导致瞒报或误诊。

**三类失败**：
- **可预防失败**：常规业务中的小流程偏差
- **复杂失败**：错误流程或系统故障
- **智慧失败**：不确定性和实验造成——这是创新的代价

**领导者必须对抗本能**：面对看似不胜任的迹象时，克制表达强烈不满的冲动。用去污名化的语言创造有利于探索、发现、反思和实验的文化。

**三步法**：
1. **检测失败**：拥抱信使、收集数据、征求反馈、奖励失败检测
2. **分析失败**：召集跨学科团队、采取系统性方法分析数据
3. **有目的实验**：奖励实验和失败、使用去污名化语言、抵消心理障碍、设计智慧实验

#### 行动4：跨越边界（Crossing Boundaries）

**核心要义**：今天成功的团队不仅围绕共享的会议桌协作，还能跨越多重边界进行合作。边界存在的根因是不同群体的人持有各自想当然的假设，思维模式多元。看似普通的行为——请求帮助、提供帮助、表达好奇、直言不讳地表达理解——能在降低协作障碍方面发挥神奇功效。

**三类边界**：
- **物理距离边界**（地点差异）：定期访问其他工作地，关注属地化独特知识，贡献知识库
- **知识边界**（组织/专业差异）：分享个人视角和专业知识的独特价值，建立集体身份认同，使用边界对象（图画、模型、原型）
- **阶层地位边界**（等级/职业地位差异）：恐惧会阻碍交流和分享，领导者需更具包容性，主动邀请成员参与对话

**领导者关键行动**：建立高阶目标、培养好奇心、提供过程指南。心理安全感使人们更容易进行跨边界沟通和实验。

### 9.2 生成规则

```
1. 结合对全场会议的分析判断，从四项领导力行动中选择最紧迫的一项
2. 基于该行动给出一条具体、可落地的建议，建议必须引用该行动的可操作步骤或策略
3. 建议必须结合会议中的具体表现，不能空泛
4. reasoning 必须综合论述：用分析语言把观察到的现象串联成论证，像资深顾问私下跟领导者聊天，不要罗列"因为A、B、C"
```

### 9.3 选择逻辑

| 会议特征 | 最可能对应的行动 | 选择理由 |
|---------|----------------|---------|
| 自我保护框架主导、目标意义模糊、角色不清 | 为学习构建框架 | 默认框架是自我保护的，必须在三个维度（领导者角色/成员角色/项目意义）上同时重构，否则后续行动缺乏基础 |
| 不敢发言、不敢承认错误、等级恐惧 | 营造心理安全感 | 四种印象风险（无知/无能/消极/破坏性）压制了自我表达，PS是组队和学习的基础设施 |
| 瞒报问题、惩罚失败、缺乏实验 | 学会从失败中学习 | 情感上无法把失败和责备剥离导致惩罚性反应→瞒报→组织失去学习机会 |
| 信息孤岛、部门壁垒、专业隔阂 | 跨越边界 | 三类边界（距离/知识/地位）阻碍信息流动和协作，看似普通的求助和好奇行为可以降低障碍 |

**优先级提示**：当多个行动都相关时，优先选择"为学习构建框架"——Edmondson明确指出"框架构建是一个好的起点"，它开启了建立心理安全环境的过程。如果框架已经到位但安全感缺失，选"营造心理安全感"。

### 9.4 字段结构

## 11. 项目阶段对分析的影响

### 11.1 阶段判断


### 11.2 启动期Behavior调整

| Behavior | 启动期调整 |
|---------|-----------|
| **speakingUp** | 降低期望，Green阈值从6分降到4分 |
| **collaboration** | 降低期望，有基本关系即可 |
| **experimentation** | **主动标记Grey**，不适用 |
| **reflection** | **主动标记Grey**，不适用 |

### 11.3 启动期与Learning Zone

```
核心矛盾与解决：
- 启动期 experimentation 和 reflection 标记为 Grey 是正常的
- Grey 在决策树中统一视为 L（缺乏证据=未发生）
- 不再对启动期的 Grey 特殊处理为 M，原因：
  - Grey=M 可能导致 E/R 均为 Grey 时决策树误判为学习区
  - 真正的 H/M 级行为应该有实际证据（Green/Blue），而非缺乏证据的 Grey
  - 如果团队确实展现了实验/反思，LLM 应给出 Green/Blue 而非 Grey

决策树中的 Grey 处理（统一规则）：
- 所有阶段：Grey → 视为 L，无证据=未发生
- 启动期和非启动期使用相同规则，不做特殊豁免

启动期 + 学习区判定：
- 如果团队确实展现了 H 级实验和反思（Green/Blue），仍可到达学习区
- 如果 E/R 为 Grey（无证据），决策树中 E/RC 会被判定为 L，自然无法进入学习区
- 不需要硬性封顶，逻辑自洽地解决冲突
```

### 11.4 启动期以后Behavior标准

```
所有维度采用标准期望：
- speakingUp：需要主动暴露问题、寻求帮助
- collaboration：需要跨职能协作、相互支持
- experimentation：需要解决方案创新
- reflection：需要迭代优化、复盘改进
```

### 12.3 核心设计原则

```
为什么用决策树而不是分数推导Zone？

决策树的优势：
- 直接从行为色标推导Zone，无中间层（PS/WS），避免信息损失
- 决策树逻辑透明：每个Zone的判定路径清晰可追溯
- 覆盖81种排列组合，MECE（互斥且完备）
- 不存在"PS不仅与S/C有关"的问题——决策树直接用S和E/R，无需假设维度归属

PS/WS 分数的角色：
- ❌ 不参与 Zone 判定（旧方案通过 PS×WS 映射表推导，已被决策树替代）
- ✅ 用于 analysis 文字书写（解读视角，PS/WS 为透镜、行为为佐证）
- ✅ 用于前端四象限图展示（显示分数对应的位置）

两步走的逻辑：
→ 步骤1：决策树从行为色标推导 Zone，一次性确定最终 Zone
→ 步骤2：一致性检查仅警告，不修正（避免修正导致循环）
```

### 12.4 qualityFlag对验证的影响

```
qualityFlag = "low_sample"：
- 提示词要求在 summary 中注明"发言样本有限"
- 行为评估中证据不足的维度标记为 Grey

qualityFlag = "unbalanced"：
- 严禁判断为学习区
- 提示词要求在 summary 中注明"发言集中于少数成员"

qualityFlag = "normal"：
- 正常分析
```

### 12.5 一致性报告输出

```
系统会在控制台输出详细的一致性检查报告：

=== 一致性检查报告 ===
🎯 [最终] Zone 由决策树决定: LLM返回="Learning", 决策树指向="Comfort" → 以决策树为准
🎯 团队基调: Comfort (Blue) - 团队氛围良好但挑战不足...
✅ 行为级别一致 [speakingUp]: Green
⚠️ 行为级别与Zone可能不一致 [experimentation]: Zone="Comfort" 期望 [Red], 但LLM分析为 "Green"
ℹ️ 低样本量模式：analysis 和 leaderAdvice 判定放宽
...
```
