## 1. 报告整体结构

报告采用“统一模板 + 结构化分析结果”的方式渲染：

| 层级 | 承载文件 | 作用 |
|------|---------|------|
| 报告模板 | `src/components/AnalysisDashboard.tsx` | 负责整体布局、姓名隐藏、PDF 导出 |
| 图表组件 | `src/components/charts/TeamStateChart.tsx`、`src/components/charts/NetworkGraph.tsx` | 负责状态四象限图与对话网络图 |
| 动态数据 | `AnalysisResult` | 来自网页上传异步任务或飞书会议转写分析 |
| 飞书报告入口 | `/report?recordId=xxx&integrationId=xxx&orgTargetId=xxx` | 读取多维表格 `JSON数据` 字段并复用同一模板 |

当前模板分为四个主部分：

```text
┌──────────────────────────────────────────────────────────────┐
│ 报告头部                                                     │
│ 标题 + 方法说明 + 会议类型 + 报告生成时间                    │
├──────────────────────────────────────────────────────────────┤
│ 第一部分：团队整体状态如何？                                  │
│ 左：Zone 四象限图   右：分段状态解读                          │
├──────────────────────────────────────────────────────────────┤
│ 第二部分：团队是如何互动的？                                  │
│ 左：NetworkGraph    右：互动分析 + 风险提醒                   │
├──────────────────────────────────────────────────────────────┤
│ 第三部分：有哪些对话没有完形？有哪些非共识值得被看见？        │
│ 未完形对话列表 + 非共识列表                                   │
├──────────────────────────────────────────────────────────────┤
│ 第四部分：领导者可以做什么？                                  │
│ 建议正文 + 为什么给出这个建议                                 │
└──────────────────────────────────────────────────────────────┘
```

说明：

- 网页上传链路会先创建 `web_analysis_tasks` 异步任务，完成后将 `AnalysisResult` 渲染到前端页面。
- 飞书链路会把 `AnalysisResult` 写入多维表格 `JSON数据` 字段，再由报告页读取并复用同一模板。
- 当前页面不再展示旧版“雷达图 + 四张行为分析卡片 + communication 列表”的布局；行为维度仍存在于 JSON 中，但主要用于分析引擎、markdown 输出和后续扩展。

## 2. 报告头部

### 2.1 固定文案

头部标题固定为：

```text
Teaming 视角下的会议动力分析报告
```

标题下方会附带一段方法说明，强调报告从以下视角观察会议：

- 团队整体状态
- 发言互动模式
- 未完形对话
- 非共识
- 领导者行动建议

### 2.2 基本信息

头部右上仅保留轻量装饰图标，不承载业务信息。核心信息展示在标题下：

| 字段 | 来源 | 说明 |
|------|------|------|
| 会议类型 | `result.metadata.meetingType` | LLM 对会议性质的概括 |
| 报告时间 | `result.reportTimestamp` | 后端生成的北京时间字符串 |

当前模板不在头部显式展示 `projectPhase`、`qualityFlag`，但这些字段仍存在于数据结构中，供分析链路与后续页面能力使用。

## 3. 第一部分：团队整体状态

### 3.1 布局

第一部分采用左右双栏：

- 左侧：`TeamStateChart` 四象限图。
- 右侧：状态解读文本卡片。

### 3.2 四象限图

四象限图以 `teamState` 为输入，展示：

- 当前 Zone
- 心理安全感分数对应的位置
- 工作标准分数对应的位置
- 当前区域高亮，其余区域淡化

图表中 Zone 的视觉映射如下：

| Zone | 中文 | 主色 |
|------|------|------|
| Learning | 学习区 | 绿色 |
| Comfort | 舒适区 | 蓝色 |
| Anxiety | 焦虑区 | 红色 |
| Apathy | 冷漠区 | 灰色 |
| Difficult to Judge | 难以判断 | 中性灰 |

### 3.3 状态解读文本

`teamState.analysis` 在新版输出中要求按三段式写作：

- `【区域判断】`
- `【工作标准】`
- `【心理安全感】`

前端会通过 `parseDiagnosisSections()` 自动拆段，并按如下方式呈现：

- 顶部固定标签：`从这场会议来看，团队目前处于{Zone中文名}`
- 第二段：`从工作标准来看`
- 第三段：`从心理安全感来看`

如果 LLM 未严格输出标签，前端仍会尝试用“从工作标准来看”“从心理安全感来看”这样的自然语言锚点做兜底拆分。

### 3.4 呈现原则

- 前端展示时会移除重复的开头引导语，避免同一句话重复出现两次。
- 文字中禁止暴露 PS/WS 的具体分数和审计规则，只保留动力学解释。
- 若开启“隐藏姓名”，这一部分中的人名也会被替换成字母代号。

## 4. 第二部分：团队互动网络

### 4.1 显示条件

只有当 `result.dialogueNetwork` 存在，且 `nodes.length > 0` 时，本部分才显示。

### 4.2 布局

第二部分同样使用左右双栏：

- 左侧：`NetworkGraph`
- 右侧：互动解读与风险提醒

### 4.3 数据来源

互动网络依赖以下结构：

```typescript
dialogueNetwork?: {
  nodes: NetworkNode[];
  edges: NetworkEdge[];
  analysis: string;
  riskAssessment: string;
}
```

其中：

- `nodes` 描述参与者及其互动角色。
- `edges` 描述谁和谁之间存在互动，以及互动强弱。
- `analysis` 描述网络整体特征。
- `riskAssessment` 描述值得注意的结构性风险。

### 4.4 发言占比

`analysisService.ts` 会在 LLM 返回后重算 `communication.speakingShare`，并同步写回 `dialogueNetwork.nodes`：

- 优先根据时间戳推算每位发言者时长。
- 时间戳不足时退回字符数占比估算。
- 最终结果四舍五入并校正到总和 100%。

因此，网络图上显示的百分比以服务端重算结果为准，而不是直接相信 LLM 原始输出。

### 4.5 文本区内容

右侧文本区包含：

- 互动解读：`dialogueNetwork.analysis`
- 风险提醒：若 `riskAssessment` 非空，则以“需要注意的是”标签单独展示

姓名隐藏模式下：

- 节点名称会被替换为字母代号
- 边的 `source/target` 会同步替换
- 右侧文本中的人名也会同步替换

## 5. 第三部分：未完形对话与非共识

### 5.1 模块作用

这一部分用于承载“会议表面看似聊完，但实际上还没闭合”的内容，以及“存在差异但未被正面看见”的内容。

它是新版模板相较旧版报告最重要的结构升级之一。

### 5.2 显示条件

满足以下任一条件时展示整块：

- `unfinishedDialogues.length > 0`
- `unseenDisagreements.length > 0`

### 5.3 未完形对话

每条未完形对话包含：

| 字段 | 说明 |
|------|------|
| `topic` | 哪个话题没有完形 |
| `whyUnfinished` | 为什么说它没有完形，卡在何处 |
| `whyNeedsClosure` | 为什么需要补完，否则会影响什么 |

前端样式为红色提示风格，用于强化“这里其实没聊完”的感觉。

### 5.4 非共识

每条非共识包含：

| 字段 | 说明 |
|------|------|
| `topic` | 非共识主题 |
| `whatEachSideSays` | 各方分别怎么说 |
| `whyItMatters` | 为什么这个差异值得被看见 |

前端呈现上：

- 话题标题置顶
- “各方的不同看法” 与 “为什么值得被看见” 分行展示

### 5.5 关键假设

`keyAssumptions` 已进入新版 `AnalysisResult` 结构，但当前 `AnalysisDashboard.tsx` 暂未单独渲染该模块。

它当前主要用于：

- `markdownFormatter.ts` 输出到飞书多维表格摘要
- 保留与 `Project 2` 分析结果结构一致性
- 为后续前端扩展预留字段

## 6. 第四部分：给领导者的建议

### 6.1 结构

第四部分固定包含两块：

- 建议正文：`leaderAdvice.advice`
- 解释说明：`leaderAdvice.reasoning`

### 6.2 展示方式

| 区块 | 样式 | 说明 |
|------|------|------|
| 建议正文 | 浅灰背景卡片 | 作为主要阅读内容 |
| 为什么给出这个建议 | 浅蓝背景说明卡片 | 用于解释判断逻辑 |

### 6.3 数据要求

`leaderAdvice` 结构如下：

```typescript
interface LeaderAdvice {
  action: 'frame_for_learning' | 'create_psychological_safety' | 'learn_from_failure' | 'cross_boundaries';
  advice: string;
  reasoning: string;
}
```

当前模板不会直接展示 `action` 的英文值，而是只展示具体建议内容和解释理由。

## 7. 图表说明

### 7.1 TeamStateChart

**位置**：第一部分左栏

**输入**：`teamState`

**语义**：

- 横轴：心理安全感
- 纵轴：工作标准
- 最终高亮区域由 `teamState.zone` 决定

说明：

- 分数只用于图上定位和视觉辅助，不用于前端再次推导 Zone。
- 前端不做二次判定，完全消费后端已修正后的结果。

### 7.2 NetworkGraph

**位置**：第二部分左栏

**输入**：`nodes` + `edges`

**节点角色**：

| `playerRole` | 中文标签 | 说明 |
|-------------|---------|------|
| `mover` | 发起者 | 提出主张、目标或行动建议 |
| `follower` | 跟随者 | 支持提议并补充、落实和推进 |
| `opposer` | 反对者 | 质疑假设、指出风险或提出不同意见 |
| `bystander` | 旁观者 | 暂时跳出立场、观察并描述整体情况 |
| `silent` | 无明显参与 | 发言极少、存在感弱 |

**连线强度**：

| `weight` | 含义 |
|---------|------|
| `strong` | 高频互动 |
| `moderate` | 中等频率互动 |
| `light` | 轻度互动 |

**附加说明**：

- 节点旁边的百分比代表发言占比。
- 连线越粗，表示互动越频繁。
- 布局为静态扇区布局，不使用力导向动画，目的是保持 PDF 导出稳定、视觉可复现。

## 8. 交互功能

### 8.1 隐藏姓名

点击“隐藏姓名”后，前端会：

- 提取 `communication`、`leaderAdvice`、`teamState.analysis` 中出现的人名
- 建立“姓名 -> 字母代号”映射
- 替换报告文本、网络图节点名、网络边的起终点名称

代号规则：

- 前 26 人使用 `A-Z`
- 超过 26 人后使用 `A1`、`B1` 这类扩展形式

### 8.2 返回

如果父组件传入 `onReset`，页面左上会显示“返回”按钮，用于回到上一步或列表页。

### 8.3 控件隐藏

`AnalysisDashboard` 支持 `hideControls`：

- 为 `true` 时，不显示返回、隐藏姓名、导出 PDF 按钮
- 常用于飞书报告页或需要纯展示的场景

## 9. PDF导出

### 9.1 导出方式

使用 `html-to-image` 的 `toJpeg()` 将整页报告渲染为 JPEG，再使用 `jsPDF` 生成 PDF。

### 9.2 关键参数

| 参数 | 值 | 作用 |
|------|----|------|
| `quality` | `0.85` | 控制 JPEG 压缩质量 |
| `pixelRatio` | `1.5` | 平衡清晰度与文件体积 |
| `skipFonts` | `true` | 降低字体加载带来的不稳定性 |
| `cacheBust` | `true` | 避免使用陈旧图片缓存 |

### 9.3 文件命名

文件名格式如下：

```text
报告-{自定义标题或未命名}-{reportTimestamp}.pdf
```

其中时间戳中的空格、冒号、连字符会被替换为下划线，确保跨平台下载兼容。

### 9.4 跨域样式处理

导出时通过 `filter` 排除非当前域名的 `<link rel="stylesheet">`：

- 避免 `html-to-image` 读取外部样式时触发跨域报错
- 保留本域样式，尽可能与页面视觉一致

## 10. 附录：数据结构

### 10.1 AnalysisResult

```typescript
interface AnalysisResult {
  reportTimestamp: string;
  metadata: MeetingMetadata;
  summary: string;
  behaviors: TeamBehaviors;
  teamState: TeamState;
  keyAssumptions: KeyAssumption[];
  unfinishedDialogues: UnfinishedDialogue[];
  unseenDisagreements: UnseenDisagreement[];
  leaderAdvice: LeaderAdvice;
  communication: CommunicationParticipant[];
  dialogueNetwork?: DialogueNetwork;
}
```

### 10.2 TeamState

```typescript
interface TeamState {
  zone: 'Apathy' | 'Comfort' | 'Anxiety' | 'Learning' | 'Difficult to Judge';
  psychologicalSafetyScore: number;
  workStandardScore: number;
  interactionFluidityScore: number;
  interactionFlowBreakdown: {
    networkStructureScore: number;
    dialogueDepthScore: number;
    crossTierInteractionScore: number;
  };
  psychologicalSafetyBreakdown: {
    speakingUpBehavior: number;
    positiveInteraction: number;
    errorTolerance: number;
  };
  workStandardBreakdown: {
    goalClarity: number;
    qualityPursuit: number;
    executionRigor: number;
  };
  analysis: string;
}
```

### 10.3 互动网络结构

```typescript
type PlayerRole = 'mover' | 'follower' | 'opposer' | 'bystander' | 'silent';

interface NetworkNode {
  name: string;
  role?: string;
  playerRole: PlayerRole;
  playerReason: string;
  speakingShare?: number;
}

interface NetworkEdge {
  source: string;
  target: string;
  weight: 'strong' | 'moderate' | 'light';
  nature?: string;
}

interface DialogueNetwork {
  nodes: NetworkNode[];
  edges: NetworkEdge[];
  analysis: string;
  riskAssessment: string;
}
```

### 10.4 未完形与非共识

```typescript
interface UnfinishedDialogue {
  topic: string;
  whyUnfinished: string;
  whyNeedsClosure: string;
}

interface UnseenDisagreement {
  topic: string;
  whatEachSideSays: string;
  whyItMatters: string;
}

interface KeyAssumption {
  assumption: string;
  whyToVerify: string;
}
```
