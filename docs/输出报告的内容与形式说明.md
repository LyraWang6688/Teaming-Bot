## 1. 报告整体结构

报告采用“固定模板 + 动态 JSON 数据”的结构：

| 部分 | 说明 |
|------|------|
| 固定模板 | `AnalysisDashboard` React 组件，负责报告布局、图表、PDF 导出 |
| 动态数据 | `AnalysisResult` JSON，来自网页上传异步任务或飞书转录稿分析 |
| 飞书报告入口 | `/report?recordId=xxx`，后端通过飞书 OpenAPI 读取多维表格 `JSON数据` 字段并渲染 |

手动上传链路先创建 `web_analysis_tasks` 异步任务，前端轮询完成后将 `AnalysisResult` 保存到页面状态；飞书链路将 `AnalysisResult` 写入多维表格 `JSON数据` 字段，报告链接只负责读取该 JSON 并复用同一模板。

```
┌─────────────────────────────────────────────────────────────┐
│                      报告头部                                │
│  会议分析报告标题 + 会议类型 + 项目阶段 + 分析时间           │
├─────────────────────────────────────────────────────────────┤
│                     组队动力综述                              │
│  一段总结性文字，描述团队整体动力状态                        │
├───────────────────────────┬─────────────────────────────────┤
│   团队整体状态定位          │      组队行为评估               │
│   - 四象限图表             │      - 雷达图                   │
│   - 动力诊断文字            │      - 4个维度标签卡片           │
├───────────────────────────┴─────────────────────────────────┤
│                      具体分析                                │
│  ┌─────────────────┐  ┌─────────────────┐                   │
│  │ SpeakingUp      │  │ Collaboration   │                   │
│  │ 分析 + 证据    │  │ 分析 + 证据     │                   │
│  ├─────────────────┤  ├─────────────────┤                   │
│  │ Experimentation│  │ Reflection      │                   │
│  │ 分析 + 证据    │  │ 分析 + 证据     │                   │
│  └─────────────────┘  └─────────────────┘                   │
├─────────────────────────────────────────────────────────────┤
│                      给领导者的建议                           │
│  建议内容 + 为什么给出这个建议                                │
└─────────────────────────────────────────────────────────────┘
```

## 2. 报告头部

### 2.1 基本信息

| 字段 | 说明 | 示例 |
|-----|------|------|
| 标题 | 固定文案 | "会议分析报告" |
| 会议类型 | LLM 从文本推断的会议性质 | "问题解决会议"、"规划讨论"、"进度同步" |
| 项目阶段 | LLM 从文本推断的项目阶段 | "启动期"、"启动期后" |

### 2.2 质量标记

当分析质量受限时会显示警告标签：

| 标记 | 含义 | 显示条件 |
|-----|------|---------|
| 样本量不足 | 有效发言数过少 | `qualityFlag === 'low_sample'` |
| 发言不均衡 | 发言分布严重不均 | `qualityFlag === 'unbalanced'` |

### 2.3 组队动力综述

位于头部底部的引用区块，包含一段 1-2 句话的总结，描述团队整体的动力状态特征。

## 3. 团队整体状态定位

### 3.1 Zone 四象限图

详见 [7.1 四象限图](#71-四象限图)

### 3.2 Zone 标签

根据团队状态显示对应的区域标签：

| Zone | 标签样式 | 含义 |
|-----|---------|------|
| Learning | 绿色 | 学习区 |
| Comfort | 蓝色 | 舒适区 |
| Anxiety | 红色 | 焦虑区 |
| Apathy | 灰色 | 冷漠区 |
| Difficult to Judge | 灰色虚线 | 难以判断 |

### 3.3 动力诊断

一段分析文字，解释为什么团队处于当前 Zone，包含：
- 心理安全感的表现
- 工作标准的体现
- 整体动力特征

> 注意：LLM 被要求在动力诊断文字中**禁止**出现任何关于分数、审计逻辑、扣分规则的字眼，仅描述观察到的团队互动状态。

## 4. 组队行为评估

### 4.1 雷达图

详见 [7.2 雷达图](#72-雷达图)

### 4.2 四维度标签卡片

| 维度 | 中文名 | 含义 |
|-----|--------|------|
| speakingUp | 直言不讳 | 坦诚直接的对话，包括提出问题、寻求反馈、讨论错误和担忧 |
| collaboration | 协同 | 协同的心态和行为，合作、相互尊重、共同目标，包括内外部协同 |
| experimentation | 实验 | 试探性、迭代的行动方式，承认不确定性，从行动结果中学习 |
| reflection | 反思 | 批判性地检视行动结果的习惯，迅速且务实，行为倾向而非正式流程 |

### 4.3 Level 级别说明

每个维度都有对应的级别标签：

| Level | 标签 | 含义 | 颜色 |
|-------|------|------|------|
| Green | 正常 | 符合该阶段的正常表现 | 绿色 |
| Blue | 良好 | 超出预期的积极表现 | 蓝色 |
| Red | 警惕 | 需要关注的消极表现 | 红色 |
| Grey | 不适用 | 该阶段不适用此维度 | 灰色 |

## 5. 具体分析详情

每个维度（speakingUp、collaboration、experimentation、reflection）都有独立的分析卡片：

### 5.1 卡片结构

```
┌─────────────────────────────────────┐
│  维度名称                    [级别]  │
├─────────────────────────────────────┤
│  分析总结文字                       │
│  （该维度的整体评估）               │
├─────────────────────────────────────┤
│  举例                               │
│  ┌─────────────────────────────┐   │
│  │ 引用1：发言内容...           │   │
│  └─────────────────────────────┘   │
│  ┌─────────────────────────────┐   │
│  │ 引用2：发言内容...           │   │
│  └─────────────────────────────┘   │
└─────────────────────────────────────┘
```

### 5.2 字段说明

| 字段 | 说明 |
|-----|------|
| 分析总结 | 对该维度行为的整体评价，包含分数范围描述 |
| 举例 | 2-3 个具体的发言引用，作为分析结论的证据 |

## 6. 给领导者的建议

基于 Edmondson "组织学习"四项领导力行动框架，结合全场会议分析，选择**最相关的一项**给出建议。

### 6.1 四项领导力行动

| 行动 | 英文 | 核心含义 | 关键要点 |
|-----|------|---------|---------|
| 为学习构建框架 | frame_for_learning | 框架即诠释，决定人们如何感知和理解环境。默认框架是自我保护的，领导者必须主动重构为学习导向 | 三维度：领导者角色（表达互赖+自身可错）、成员角色（精挑细选的专家）、项目意义（有说服力+有挑战但不过度） |
| 营造心理安全感 | create_psychological_safety | 以人际信任和相互尊重为底色的氛围，关注人际关系风险后果的感知。PS不是做"老好人"，不是降低标准 | 四种印象风险：被视为无知、无能、消极、有破坏性；七项好处：直言、清晰思维、建设性冲突、缓冲失败、促进创新、消除高绩效障碍、增强责任感 |
| 学会从失败中学习 | learn_from_failure | 三类失败：可预防（流程偏差）、复杂（系统故障）、智慧（不确定性+实验）。领导者须对抗本能，克制面对看似不胜任时表达不满的冲动 | 三步法：检测失败（拥抱信使、奖励检测）→ 分析失败（跨学科团队、系统方法）→ 有目的实验（去污名化语言、设计智慧实验） |
| 跨越边界 | cross_boundaries | 三类边界：物理距离、知识（组织/专业差异）、阶层地位（等级差异）。边界存在的根因是不同群体持有各自想当然的假设 | 普通行为的神奇功效：请求帮助、提供帮助、表达好奇、直言不讳地表达理解；领导行动：建立高阶目标、培养好奇心、提供过程指南 |

### 6.2 结构

```
┌─────────────────────────────────────────────────────────────┐
│  [图标]  给领导者的建议                                      │
├─────────────────────────────────────────────────────────────┤
│  建议内容                                                    │
│  关注方向（让领导者意识到值得关注的点）+ 举例（可能的具体做法）│
│                                                              │
│  为什么给出这个建议                                           │
│  综合论述，用分析语言串联观察到的现象                         │
└─────────────────────────────────────────────────────────────┘
```

### 6.3 选择逻辑

- **焦虑区**：优先选择"营造心理安全感"——团队缺乏人际信任和相互尊重的底色，成员面临四种印象风险（被视为无知/无能/消极/有破坏性），不敢说不敢试不敢求助。领导者需要贴近员工、承认知识局限、展示可错面、把失败当学习机会
- **舒适区**：优先选择"为学习构建框架"——团队自我保护的框架已经固化，"一直这么做挺好的"。需要领导者重构框架，从三个维度同时入手：表达互赖和自身可错、强调成员是精挑细选的专家、传递有挑战但不过度的项目意义
- **冷漠区**：优先选择"为学习构建框架"——团队缺乏方向和目标感，精力用于争权夺利而非共同目标。需要领导者明确传递项目意义，让成员看到自己与成功结果的互赖关系
- **学习区**：优先选择"跨越边界"——已具备安全感和工作标准，下一步是跨越物理距离、知识、阶层地位三类边界，通过请求帮助、表达好奇、直言不讳等普通行为降低协作障碍

> 以上为倾向性指引，最终由 LLM 根据会议实际内容判断选择哪项行动。

## 7. 图表说明

### 7.1 四象限图

**位置**：报告左侧，"团队整体状态定位"区域

**结构**：

```
        工作标准高
            ↑
            │
   焦虑区    │    学习区
   (红色)    │    (绿色)
 ────────────┼─────────────→ 心理安全感高
   冷漠区    │    舒适区
   (灰色)    │    (蓝色)
            │
            ↓
        工作标准低
```

**评分说明**：
- X 轴：心理安全感（0-10 分）
- Y 轴：工作标准（0-10 分）
- 分数 > 6 定义为"高"，≤ 6 定义为"低"
- **注意**：分数用于四象限图的定位参考和 analysis 文字书写，最终 Zone 由决策树从行为色标推导（覆盖 81 种排列组合，保证 MECE）
- **Difficult to Judge 触发条件**：有效发言 < 10 句，或 ≥50% 行为维度为 Grey → 强制返回"难以判断"
- **分数说明**：PS/WS 分数仅作为辅助参考展示，不参与 Zone 判定。已废除"≥3项信号→分数≤4分"的强制限制

**当前 Zone 区域**会高亮显示，其他区域淡化。

### 7.2 雷达图

**位置**：报告右侧，"组队行为评估"区域

**结构**：

```
           直言不讳
              ↑
              │
      实验     /‾‾‾\     反思
              ─┼─
              │
              ↓
           协同
```

**数值范围**：
- 0-5 分：警惕区域（红色）
- 5-7 分：良好区域（蓝色）
- 7-10 分：健康区域（绿色）

**图例**（底部居中）：
- 绿色圆点：健康区域
- 蓝色圆点：良好区域
- 红色圆点：警惕区域

**轴标签**：仅显示维度名称（直言不讳/协同/实验/反思），**不显示具体分数数字**。

**低分兜底**：当某维度分数为 0 或极低时，雷达图使用 `MIN_VISIBLE = 0.3` 作为最小显示值，确保多边形面积始终可见，避免退化为线段或点。但数据点标记（圆点）始终反映实际分数位置。

**数据点标记**：每个维度顶点显示紫色实心圆点（白色描边），即使面积很小也能看到各维度的实际位置。

## 8. 交互功能

### 8.1 隐藏/显示姓名

点击"隐藏姓名"按钮，可将报告中所有发言者姓名替换为字母（A、B、C...）。

**隐藏范围**：
- 给领导者的建议中提到的发言者姓名
- 发言引用中的被提及者姓名
- 分析文字中的人名

**实现方式**：前端使用 `useMemo` + 正则匹配提取所有人名，建立姓名→字母映射表，在渲染时替换。

### 8.2 返回列表

点击"返回"按钮，返回到报告列表页面。

### 8.3 文件上传

**支持格式**：`.txt` 和 `.docx`

**上传方式**：
- 点击"添加文件"按钮选择文件
- 拖拽文件到上传区域

**自动提交**：选择或拖入文件后**立即开始分析**，无需手动点击提交按钮。分析期间显示加载动画。

## 9. PDF导出

### 9.1 导出方式

使用 `html-to-image` 库将报告 DOM 渲染为图片，再使用 `jsPDF` 生成 PDF 文件。

### 9.2 导出格式

截图使用 **JPEG 格式**（而非 PNG），以压缩体积：

| 参数 | 值 | 说明 |
|-----|-----|------|
| 截图格式 | JPEG | 有损压缩，体积远小于 PNG |
| quality | 0.85 | JPEG 压缩质量（0-1） |
| pixelRatio | 1.5 | 截图像素比（默认2x，降低以减小体积） |
| PDF 嵌入格式 | JPEG + FAST 压缩 | jsPDF 内部存储格式 |

**预估体积**：每份报告约 3-8MB（原 PNG 方案约 40-80MB）。

### 9.3 文件命名

```
报告-{会议标题}-{时间戳}.pdf
```

示例：`报告-问题解决会议-2024-01-15_14-30-00.pdf`

### 9.4 导出特点

- 保持与网页一致的视觉效果
- 包含所有图表和数据
- 字体大小与网页显示一致
- 批量下载与单独下载使用相同的渲染组件，确保视觉一致

### 9.5 批量下载

在报告列表页面，当有已完成的报告时，可一键下载所有报告为 ZIP 文件。

**实现方式**：
- 使用 React `createRoot` API 动态渲染完整的 `AnalysisDashboard` 组件
- 逐个截图并生成 PDF，最后打包为 ZIP
- 渲染时等待字体加载完成，确保与单独下载一致

### 9.6 跨域样式处理

PDF 导出时使用 `html-to-image` 的 `filter` 选项排除外部样式表链接，避免 CSS 跨域访问错误，只保留同域样式表。

## 10. 附录：数据结构参考

### AnalysisResult

```typescript
interface AnalysisResult {
  reportTimestamp: string;        // 报告生成时间
  metadata: MeetingMetadata;      // 元数据
  summary: string;               // 组队动力综述
  behaviors: TeamBehaviors;      // 4个行为维度
  teamState: TeamState;          // 团队状态
  leaderAdvice: LeaderAdvice;     // 给领导者的建议
  communication: CommunicationParticipant[]; // 沟通数据
}
```

### TeamState

```typescript
interface TeamState {
  zone: 'Apathy' | 'Comfort' | 'Anxiety' | 'Learning' | 'Difficult to Judge';
  psychologicalSafetyScore: number;    // 心理安全感 (0-10)
  workStandardScore: number;          // 工作标准 (0-10)
  interactionFluidityScore: number;   // 互动流畅度 (0-10)
  psychologicalSafetyBreakdown: {
    speakingUpBehavior: number;
    positiveInteraction: number;
    errorTolerance: number;
  };
  workStandardBreakdown: {
    goalClarity: number;
    qualityPursuit: number;
    executionRigor: number;
  };
  interactionFlowBreakdown: {
    networkStructureScore: number;
    dialogueDepthScore: number;
    crossTierInteractionScore: number;
  };
  analysis: string;                   // 动力诊断文字（Zone被修正时由后端模板重新生成，末尾可能含⚠️修正注释）
}
```

### BehaviorMetric

```typescript
interface BehaviorMetric {
  level: 'Red' | 'Blue' | 'Green' | 'Grey'; // 级别
  score: number;                              // 分数 (0-10)
  evidence: string[];                         // 证据引用
  summary: string;                            // 分析总结
}
```

### LeaderAdvice

```typescript
interface LeaderAdvice {
  action: 'frame_for_learning' | 'create_psychological_safety' | 'learn_from_failure' | 'cross_boundaries';
  advice: string;      // 建议内容
  reasoning: string;    // 为什么给出这个建议：综合论述
}
```

### Quote

```typescript
interface Quote {
  speaker: string;   // 发言者姓名
  text: string;      // 发言原话
}
```

### BatchItem

```typescript
interface BatchItem {
  id: string;
  file: File;
  status: 'PENDING' | 'ANALYZING' | 'COMPLETE' | 'ERROR';
  taskId?: string;
  result?: AnalysisResult;
  error?: string;
}
```
