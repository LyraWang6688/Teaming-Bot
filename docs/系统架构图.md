# 系统架构图

> 最后更新时间：2026-06-18

本文档基于当前仓库代码整理项目的四类图：

- 框架图
- 流程图
- 业务图
- 数据流图

当前项目保留两条业务入口：

- 手动入口：前端上传 `.txt/.docx` 文件，调用 `/api/analyze`
- 自动入口：飞书 `Webhook + OpenAPI` 自动接收事件并触发分析

两条入口共享同一套分析内核：`analysisService.ts + 豆包 Provider + Teaming/Zone 规则 + AnalysisDashboard`

- 核心分析服务统一走豆包大模型，不区分手动入口和飞书入口。
- 飞书链路通过 OpenAPI 获取令牌、搜索妙记、导出文字稿并读写 Base 记录。

## 二、流程图

```mermaid
flowchart TD
    Start["开始"] --> Entry{"从哪里进入?"}

    Entry -->|"手动上传"| Upload["前端选择 .txt/.docx\nInputForm.tsx"]
    Entry -->|"飞书事件"| Event["收到飞书 Webhook 事件\nwebhook/route.ts"]

    Upload --> AnalyzeReq["POST /api/analyze"]
    AnalyzeReq --> Parse["解析输入\n.txt 直接读取\n.docx 用 mammoth 提取文本"]
    Parse --> AnalyzeCore["analyzeMeetingText()"]

    Event --> Verify["验签 / challenge / 去重"]
    Verify --> EventType{"事件类型?"}
    EventType -->|"会议结束"| Waiting["写入/更新 Base\n状态=会议结束"]
    EventType -->|"其他事件"| Ignore["忽略"]
    Waiting --> Search1["按标题/组织者/时间窗搜索妙记"]
    Search1 --> Found{"是否找到妙记?"}
    Found -->|"否"| Retry["第 2/3/4/5 分钟继续搜索"]
    Found -->|"是"| FetchDoc["导出妙记文字稿"]
    Retry --> RetryDone{"5 分钟内仍未找到?"}
    RetryDone -->|"是"| Fail1["写回失败状态 + 错误信息"]
    RetryDone -->|"否"| Search1
    FetchDoc --> SaveTranscript["状态=获取文字稿中\n写入会议文字稿"]
    SaveTranscript --> AnalyzeCore

    AnalyzeCore --> Prompt["构造 Teaming 提示词\nteamingRules.ts"]
    Prompt --> Doubao["调用豆包模型"]
    Doubao --> ParseJson["解析/修复 JSON"]
    ParseJson --> ZoneFix["按行为决策树修正 Zone\nzoneConfig.ts"]
    ZoneFix --> Format["补 reportTimestamp\n输出 AnalysisResult"]

    Format --> ReturnManual{"来源是手动上传?"}
    ReturnManual -->|"是"| ManualDone["前端展示 AnalysisDashboard\n支持 PDF 导出"]
    ReturnManual -->|"否"| SaveBase["状态=分析中/已完成\n写回摘要、JSON、报告链接"]
    SaveBase --> ReportLink["/report?recordId=..."]
    ReportLink --> ReportPage["报告页读取 Base\n渲染 AnalysisDashboard"]

    Fail1 --> Done["结束"]
    Ignore --> Done
    ManualDone --> Done
    ReportPage --> Done
```

### 说明

- 手动链路直接以上传文件为起点，调用 `/api/analyze`。
- 飞书链路当前统一由会议结束事件驱动。
- 真正耗时的步骤包括：搜索妙记、导出文字稿、调用豆包分析、写回 Base，全部由 `webhookProcessor.ts` 异步执行。
- 两条链路在 `analyzeMeetingText()` 汇合，之后共享相同的分析流程。

## 三、业务图

```mermaid
flowchart LR
    User["用户 / 管理者"] --> LocalUpload["本地会议纪要上传"]
    User --> FeishuMeeting["飞书会议事件触发"]
    User --> ReportView["查看报告 / 下载 PDF"]

    LocalUpload --> MeetingText["会议文本"]
    FeishuMeeting --> MeetingMeta["会议基本信息"]
    FeishuMeeting --> MinuteSearch["妙记搜索条件"]

    MeetingText --> AnalysisEngine["Teaming 动力分析引擎"]
    MeetingMeta --> AnalysisEngine
    MinuteSearch --> TranscriptContent["妙记文字稿内容"]
    TranscriptContent --> AnalysisEngine

    AnalysisEngine --> Summary["会议总结"]
    AnalysisEngine --> Behaviors["四项行为评估\nSpeaking Up / Collaboration / Experimentation / Reflection"]
    AnalysisEngine --> TeamState["团队状态\nApathy / Comfort / Anxiety / Learning"]
    AnalysisEngine --> LeaderAdvice["给领导者的建议"]
    AnalysisEngine --> Communication["沟通参与度数据"]

    Summary --> Report["报告"]
    Behaviors --> Report
    TeamState --> Report
    LeaderAdvice --> Report
    Communication --> Report

    Report --> Base["飞书多维表格记录"]
    Report --> Dashboard["前端报告仪表盘"]
    Dashboard --> PDF["PDF 导出"]
```

### 说明

- 业务上系统服务两个主要场景：人工上传分析、飞书自动分析。
- 不论入口如何，最终都抽象为“会议文本 -> 团队行为诊断 -> 团队状态评估 -> 报告输出”。
- 报告既可在前端页面直接查看，也可回写飞书多维表格，并通过 `recordId` 打开详情页。

## 四、数据流图

```mermaid
flowchart TD
    F1["原始输入数据\nFile(.txt/.docx) 或 Feishu Event JSON"] --> T1["输入解析层"]
    T1 --> D1["纯文本会议纪要 string"]

    D1 --> T2["分析请求构造"]
    T2 --> D2["LLM Messages\nsystem prompt + user prompt + JSON schema"]

    D2 --> T3["豆包 API"]
    T3 --> D3["LLM 原始返回 string(JSON/半结构化)"]

    D3 --> T4["解析与修复"]
    T4 --> D4["LLMResponse\nmetadata/summary/behaviors/teamState/leaderAdvice/communication"]

    D4 --> T5["后处理校正"]
    T5 --> D5["AnalysisResult\n+ reportTimestamp\n+ Zone corrected"]

    D5 --> T6{"输出去哪里?"}
    T6 -->|"手动入口"| O1["HTTP JSON 响应\n/api/analyze -> 前端状态列表"]
    T6 -->|"飞书入口"| O2["JSON.stringify(AnalysisResult)\n写入 Base 的 JSON数据 字段"]

    O2 --> T7["Base 记录结构"]
    T7 --> D6["会议ID\n会议主题\n开始时间\n结束时间\n组织者\n处理状态\n会议文字稿\n分析摘要\n报告链接\nJSON数据\n错误信息"]

    D6 --> T8["/api/feishu/record"]
    T8 --> D7["FeishuMeetingRecord\nanalysisData/processStatus/topic/..."]

    D7 --> T9["/report?recordId=..."]
    T9 --> O3["AnalysisDashboard 渲染\n图表 + 建议 + PDF"]
```

### 说明

- 原始输入数据在进入统一分析服务前，都会先转成纯文本会议纪要。
- 大模型返回的原始字符串会被解析、修复，并补全为标准 `AnalysisResult`。
- 飞书链路下，`AnalysisResult` 会被序列化后写入 Base 的 `JSON数据` 字段。
- 报告页通过 `recordId` 回读 Base 记录，再恢复为可渲染的 `analysisData`。
