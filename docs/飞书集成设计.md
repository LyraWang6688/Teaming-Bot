# 飞书集成设计文档

> 最后更新时间：2026-06-30

## 一、概述

当前飞书集成主链路以服务器内置的飞书 CLI 为核心。用户只需要在 `/feishu-config` 页面点击按钮完成初始化、授权、事件校验和多维表格初始化，不需要进入飞书后台手动创建应用、配置权限或订阅事件。

目标闭环：

1. 后端通过 `lark-cli config init --new` 为用户创建或绑定独立 CLI profile。
2. 后端通过 `lark-cli auth login --scope ... --no-wait --json` 发起用户权限授权。
3. 后端通过 `lark-cli auth login --device-code ... --json` 轮询授权结果。
4. 后端通过 `lark-cli event consume minutes.minute.generated_v1 --as user` 监听妙记生成事件。
5. 服务端从事件体读取 `event.minute_source.source_entity_id` 作为会议 ID，写入多维表格。
6. 服务端从事件体读取 `event.minute_token`，用于导出妙记文字稿。
7. 服务端调用豆包分析，并把文字稿、摘要、报告链接和 JSON 数据回写多维表格。

## 二、技术架构

```text
前端 /feishu-config
  ↓
Next.js API
  ↓
lark-cli profile 初始化 / 用户授权 / 事件监听 / OpenAPI 调用
  ↓
minutes.minute.generated_v1
  ↓
meeting_pipeline_tasks
  ↓
meetingPipelineWorker.ts
  ↓
minute_token -> transcript -> analysisService.ts -> bitableOpenApi.ts
  ↓
飞书多维表格 + 报告页
```

关键原则：

- 服务器或 Docker 容器内安装飞书 CLI，业务用户本地不安装 CLI。
- 每个用户使用独立 `profileName`，以 `userId + integrationId + profileName` 做多租户隔离。
- 所有飞书资源访问均走用户身份：`lark-cli --profile <profile> api ... --as user --format json`。
- 数据库不保存可直接访问飞书资源的真实 Bearer token；CLI 自行管理授权凭据。

## 三、事件与权限

正式事件：

| 事件名称 | 事件 ID | 用途 |
| --- | --- | --- |
| 妙记生成 | `minutes.minute.generated_v1` | 自动分析链路的唯一事件入口 |

用户授权 scope：

| 权限 ID | 用途 |
| --- | --- |
| `minutes:minutes.basic:read` | 获取妙记基础信息，并满足妙记生成事件订阅要求 |
| `minutes:minutes.transcript:export` | 导出妙记转写文字 |
| `bitable:app` | 创建、读取和编辑用户授权范围内的多维表格 |
| `offline_access` | 允许 CLI 持续访问用户已授权数据 |

## 四、前端配置页职责

页面位置：`/feishu-config`

初始化向导：

1. 创建飞书应用：后端执行 `lark-cli config init --new`，并返回 CLI 提供的验证地址。
2. 用户授权：后端执行 `lark-cli auth login --scope "<scope>" --no-wait --json`，前端展示授权入口。
3. 授权轮询：后端执行 `lark-cli auth login --device-code <device_code> --json`，成功后绑定真实飞书用户身份。
4. 事件校验：后端启动 `event consume` 并等待 ready marker，确认 `minutes.minute.generated_v1` 可监听。
5. 初始化多维表格：后端用 CLI 用户身份调用 Base API 创建或复用表格，并补齐字段。

## 五、多维表格字段

Base 初始化会自动补齐以下字段：

| 字段名 | 类型 | 说明 |
| --- | --- | --- |
| 会议ID | 文本 | 来自 `event.minute_source.source_entity_id` |
| 处理状态 | 单选/文本 | 当前管线阶段 |
| 会议文字稿 | 长文本 | 由 `minute_token` 导出的妙记文字稿 |
| 分析摘要 | 长文本 | 豆包分析摘要 |
| 报告链接 | URL | `/report?recordId=...&integrationId=...` |
| JSON数据 | 长文本 | 完整结构化分析结果 |
| 错误信息 | 文本 | 异常原因 |

## 六、会议分析流程

1. 飞书 CLI 监听到 `minutes.minute.generated_v1`。
2. `eventListenerManager.ts` 按 integration/profile 分发事件。
3. `meetingPipelineProcessor.ts` 提取：
   - `event.minute_token` 作为妙记 token。
   - `event.minute_source.source_entity_id` 作为会议 ID。
4. 系统写入或复用 `meeting_pipeline_tasks`，以 `integrationId + meetingId` 做幂等键。
5. Worker 使用 `minute_token` 导出文字稿。
6. `analysisService.ts` 调用豆包生成结构化分析。
7. `bitableOpenApi.ts` 通过 CLI 用户身份回写 Base。
8. 用户通过报告链接查看结果。

## 七、可靠性与日志

- 事件监听必须等待 CLI ready marker 后才标记事件校验成功。
- 同一 `integrationId + meetingId` 在单进程内有处理锁，避免重复导出和重复分析。
- 服务启动时会恢复 `meeting_pipeline_tasks` 中未完成的任务。
- 飞书 CLI 初始化、授权、事件监听、OpenAPI 调用和管线处理均输出结构化日志，日志字段包含 `integrationId`、`profileName`、`stage`、`durationMs` 和错误上下文。
- 日志不得打印完整 token、secret 或其他可直接访问飞书资源的凭据。

## 八、部署注意事项

- Docker 镜像内需要安装 `@larksuite/cli`，并保证 `lark-cli` 在运行时 PATH 中可用。
- CLI 配置目录默认使用 `/app/.lark-cli`，可通过 `LARKSUITE_CLI_CONFIG_DIR` 覆盖。
- 数据库 schema 变更后执行 `pnpm db:push` 同步 Supabase。
- 项目使用 pnpm，不提交 `package-lock.json`。
- 生产启动入口使用 `next start`，不使用自定义 `server.ts`。
