# 飞书集成设计文档

> 最后更新时间：2026-06-26

## 一、概述

当前飞书集成已经完成从 `Webhook + OpenAPI + 服务器环境变量` 到 `Webhook + OpenAPI + Supabase(Auth/Postgres) + 用户级集成配置` 的主链路迁移。

目标闭环：

- 接收会议结束事件
- 将会议基础信息写入多维表格
- 获取会议录制文件并解析妙记 token
- 导出妙记文字稿
- 调用豆包分析
- 回写多维表格
- 打开报告页

## 二、技术架构

### 2.1 当前正式方案

当前生产链路统一使用 Webhook 接收 `vc.meeting.participant_meeting_ended_v1`，再由服务端通过 OpenAPI 获取会议录制文件、解析 `minute_token`、导出文字稿，并写入多维表格。当前已引入数据库任务表承接会议处理状态：Webhook 会先落库到 `meeting_pipeline_tasks`，再由后台任务执行器轮询到期任务，继续执行录制、转写、分析和回写；其中 Webhook、Base 与任务恢复继续使用 `tenant_access_token`，录制文件与妙记文字稿在单 owner 模式下优先使用 `user_access_token`，失败时再回退到 `tenant_access_token`。

```text
飞书事件
  ↓
/api/feishu/webhook
  ↓
meeting_pipeline_tasks
  ↓
meetingPipelineWorker.ts
  ↓
recording / transcript / analysisService.ts / bitableOpenApi.ts
  ↓
多维表格 + 报告页
```

### 2.2 为什么使用会议结束事件

当前自动化链路以“用户参与的会议已经结束”为起点。事件体中直接提供 `meeting.id`、`topic`、`start_time`、`end_time`、`owner.id.open_id`、`host_user.id.open_id`，足以完成建档与后续录制文件轮询准备。

### 2.3 新的多租户目标架构

平台级配置继续保留在全局 `.env` 中：

- `PROJECT_PUBLIC_URL`
- 大模型配置
- `DATABASE_URL`
- `NEXT_PUBLIC_SUPABASE_URL`
- `NEXT_PUBLIC_SUPABASE_ANON_KEY`
- `SUPABASE_SERVICE_ROLE_KEY`
- `APP_ENCRYPTION_KEY`

用户级飞书配置迁移到数据库：

- 飞书应用配置：`App ID / App Secret / Webhook Verification Token`
- Base 配置：`Base App Token / Table ID / 初始化状态`
- OAuth 授权：`access_token / refresh_token / expires_at`
- 检查状态：权限、事件订阅、Webhook、OAuth、Base
- 审计日志、会议主索引与会议任务状态

## 五、前端配置页职责

页面位置：`/feishu-config`

配置页目标升级为 4 步初始化向导：

1. 登录平台账号
2. 配置飞书应用
3. 完成 OAuth 授权
4. 初始化多维表格

当前已落地的前端行为：

- 未登录用户访问 `/feishu-config` 时，只显示说明与登录入口；保存动作要求先完成 `Supabase Auth` 登录
- 登录后页面会调用 `/api/auth/me` 识别当前用户，再读取 `/api/feishu/integrations` 获取该用户当前唯一一套主集成；若尚未创建，则停留在空表单状态等待用户填写基础凭证后再创建
- 页面当前按“单账号一套主集成”组织，不再暴露多集成切换入口；基础凭证首次提交时会真实 `POST /api/feishu/integrations` 创建集成，后续更新再走 `PATCH`
- 已保存的 `App Secret`、`Webhook Verification Token`、`Base App Token` 仅以脱敏形式回显，前端不再接收密钥明文
- 选中某个集成后，页面通过 `/api/feishu/integrations/[integrationId]` 读取该集成的 OAuth 摘要、检查状态、必需权限与必需事件
- 第 2 步固定展示两部分：先保存 `App ID / App Secret / Webhook Verification Token`，再按“事件与回调 -> 权限管理 -> OAuth 重定向 URL -> 发布提示 -> 通信状态”的顺序指导用户到飞书开放平台补齐配置；页面会根据数据库中的 `lastWebhookReceivedAt / webhookStatus / eventSubscriptionStatus / authorization` 等真实状态自动判断第二阶段是否完成
- 当前向导改为左侧纵向步骤栏 + 右侧内容卡片；步骤状态会随着用户推进变色，左侧 `sticky` 状态卡会持续显示“当前在哪一步、下一步做什么”
- 第 2 步不再展示“集成名称”“附加配置”与“保存附加配置”按钮，普通用户只填写基础凭证；OAuth 回调地址直接在第 2 步展示，不再只放在授权步骤中
- 点击“授权当前集成”时，页面使用 `/api/feishu/oauth/start?integrationId=...` 发起与当前集成绑定的 OAuth；授权成功后服务端会直接回跳 `/feishu-config?oauth=success`，前端展示轻提示并刷新集成状态
- 点击“一键初始化 Base”时，页面会调用 `/api/feishu/integrations/[integrationId]/base/initialize`，自动创建 Base、复用或补齐会议信息表字段，并把 `Base App Token / Meeting Table ID / initializedAt` 回写到当前集成
- Base 初始化完成后，页面会基于当前集成返回的 `baseUrl` 直接展示“打开多维表格”链接，用户可直接进入已创建的飞书多维表格
- 前端不再把“真实检查”单独展示成第 5 步；当用户完成第 4 步后，系统会自动在后台触发真实检查，并把结果回写 `feishu_integration_checks`

当前阶段的页面边界：

- “真实检查” 当前已支持应用凭证、OAuth、Webhook/Base 联通的在线校验；`权限` 状态会基于 `tenant credentials + oauth user + bitable access` 综合推断，会议录制与妙记导出权限仍会在首条真实会议链路中继续验证
- `事件订阅` 当前通过“是否已经收到 Webhook challenge / 真实事件回调”间接判定，尚未直接读取飞书开放平台后台的订阅配置
- 页面当前明确聚焦“每个用户维护自己的 1 套主集成”；数据库结构仍保留扩展到多集成的空间，但现阶段 UI 不再提供多集成管理入口
- 系统内部校验结果当前只以“配置完成 / 自动校验中”方式反馈给用户，不再在前端展开为一整步明细状态面板

用户负责：

- 在飞书开放平台创建企业自建应用
- 开权限、配事件、配置 Webhook/OAuth 回调
- 点击 OAuth 授权

系统负责：

- 存储用户级飞书集成配置
- 使用内部状态模型做真实 API 检查
- 自动创建 Base 和字段
- 自动保存 OAuth token 并续期
- 记录初始化状态与审计日志
- 运行会议分析自动化链路

## 六、多维表格结构

当前运行时依赖一张会议信息表，字段如下：

| 字段名 | 字段类型 | 用途 |
|-------|---------|------|
| 会议ID | 文本 | 飞书会议唯一标识 |
| 会议主题 | 文本 | 会议标题 |
| 开始时间 | 日期时间 | 会议开始时间 |
| 结束时间 | 日期时间 | 会议结束时间 |
| 组织者 | 文本 | 会议拥有者或主持人的 open_id |
| 处理状态 | 单选 | 会议结束 / 获取录制文件中 / 获取文字稿中 / 分析中 / 已完成 / 失败 |
| 会议文字稿 | 文本 | 妙记转写全文 |
| 分析摘要 | 文本 | LLM 分析摘要 |
| 报告链接 | URL | `/report?recordId=xxx` 报告页面链接；写入时需使用 `{ text, link }` 结构 |
| JSON数据 | 文本 | 完整 AnalysisResult JSON |
| 错误信息 | 文本 | 异常原因，用于排障 |

## 七、完整流程

### 7.1 初始化流程

1. 用户登录平台（`Supabase Auth`）。
2. 在 `/feishu-config` 中填写并保存 `App ID / App Secret / Webhook Verification Token`，系统为当前用户创建或更新唯一一套飞书集成配置。
3. 用户在飞书开放平台中补齐事件订阅、权限和回调地址：
   - Webhook：`/api/feishu/webhook`
   - OAuth 回调：`/api/feishu/oauth/callback`
4. 飞书在保存 Webhook 地址时会先发送 `url_verification`；系统通过 `Webhook Verification Token -> integration` 命中对应集成，并在成功返回 challenge 后回写 `lastWebhookReceivedAt / webhookStatus / eventSubscriptionStatus`。页面据此自动推进第 2 步状态。
5. 用户点击“授权飞书”，系统通过 `state -> user_id + integration_id` 绑定回调结果，并将 OAuth token 直接写入数据库；授权成功后会直接跳回 `/feishu-config?oauth=success`。
6. 用户点击“一键初始化 Base”后，系统会自动创建 Base App（或复用现有 Base）、定位/创建会议信息表，并补齐运行时依赖字段。
7. 系统把 `Base App Token / Table ID / initializedAt` 写回该用户的集成配置，并自动触发真实检查，在线验证应用凭证、OAuth、Webhook 和 Base 联通情况。
8. 当所有状态通过后，前端提示“配置完成，可以开始使用系统”，系统开始自动监听和分析后续会议。

### 7.2 会议分析流程

1. 飞书发送 `vc.meeting.participant_meeting_ended_v1`。
2. `/api/feishu/webhook` 先通过 `Webhook Verification Token -> integration` 识别当前集成，再完成验签并把会议任务写入 `meeting_pipeline_tasks`。
3. 服务端从事件体中直接提取 `meeting.id`、`topic`、`start_time`、`end_time`、`owner.id.open_id`；若 `owner.id.open_id` 缺失，则回退到 `host_user.id.open_id`，再回退到 `operator.id.open_id`。
4. `meetingPipelineWorker.ts` 会轮询领取 `pending / scheduled / running` 且到期的任务，并沿用该条 `integrationContext` 继续执行后续录制获取、妙记导出、Base 读写和报告链接生成，不再依赖全局 `FEISHU_*` 配置；服务端先把 `会议ID`、`会议主题`、`开始时间`、`结束时间`、`组织者` 写入该集成自己的多维表格，并将状态写为 `会议结束`。
5. 首次建档时以 `meeting.id` 作为幂等键防止重复创建；一旦 Base 返回 `recordId`，后续状态更新、恢复链路和继续处理都优先使用 `recordId` 命中同一条记录。
6. 服务端在会议结束后第 `2`、`3`、`4`、`5` 分钟各执行一次轮询尝试，先调用 `GET /open-apis/vc/v1/meetings/{meeting_id}/recording` 获取录制文件，状态变更为 `获取录制文件中`。该接口会优先使用当前集成 OAuth 授权得到的 `user_access_token`，失败后再回退到该集成自己的 `tenant_access_token`。
7. 若接口返回 `recording.url`，服务端从 URL 最后一段解析出 `minute_token`；若录制仍在生成中，则继续等待下一轮窗口。
8. 服务端调用 `minutes/v1/minutes/:minute_token/transcript?need_speaker=true&need_timestamp=true&file_format=txt` 导出文字稿，状态变更为 `获取文字稿中`；该接口同样会优先使用当前集成 OAuth 用户身份，失败后再回退到该集成 tenant 身份；若妙记转写尚未完成，则继续等待下一轮窗口。
9. 调用豆包 API 分析，状态变更为 `分析中`；若豆包返回临时性错误，会在服务内做最多 3 次短间隔重试。
10. 回写 `会议文字稿`、`分析摘要`、`报告链接`、`JSON数据`，状态变更为 `已完成`。其中 `报告链接` 若为多维表格超链接字段，必须按 `{ text: 会议主题, link: 报告URL }` 结构写入，不能直接传普通字符串，否则可能触发 `URLFieldConvFail`。
11. 用户通过 `/report?recordId=...&integrationId=...` 打开报告页，报告接口会按 `integrationId` 回到对应集成的 Base 读取该条记录。

### 7.3 可靠性机制

1. `获取录制文件` 与 `导出妙记文字记录` 会优先使用当前集成 OAuth 授权得到的 `user_access_token`；若未完成 OAuth、token 已失效或 user 身份调用失败，则自动回退到同一集成下的 `tenant_access_token`。
2. 若第 2/3/4/5 分钟窗口内录制文件仍未生成，系统不会提前写成 `失败`，而是继续等待下一次窗口；若 5 分钟窗口结束仍无结果，则写入业务可理解的失败原因。
3. 若导出文字稿时飞书返回 `minute not ready` 或临时性 5xx/429 错误，系统会继续等待下一次窗口；若所有窗口耗尽仍未就绪，则写回失败状态。
4. 单进程内对同一会议持有处理锁，避免同一会议被并发导出、并发分析；锁超时后允许后续事件重新进入链路，防止记录长期卡住。
5. Base 回读时会统一把文本单元格和富文本片段数组归一化为纯字符串，避免启动恢复时因字段返回格式差异读不出 `meetingId`、`会议主题` 或 `组织者`。
6. Webhook 处理链路输出结构化监控日志，统一记录 `meetingId`、`recordId`、`attempt`、`organizerSource`、`minuteToken`、错误类型和重试行为，便于在应用日志中追踪录制文件、文字稿、分析和恢复过程。
7. 服务启动时会先拉起 `meetingPipelineWorker.ts`，该执行器按固定轮询间隔从数据库领取 `pending / scheduled / running` 且已到期的任务；若任务已绑定 `base_record_id`，则优先回读对应 Base 记录继续恢复；若任务尚未拿到 Base 记录，则直接使用任务 payload 中保存的会议上下文继续“获取录制文件 / 导出文字稿 / 分析”主流程。
8. 启动恢复默认开启，可通过环境变量 `FEISHU_ENABLE_STARTUP_RECOVERY=false` 显式关闭；当前版本已把“Webhook 接收”和“任务执行”拆成两个模块，并通过数据库任务表 + `SKIP LOCKED` 方式领取到期任务，后续若进入更高并发生产阶段，可继续把同一套执行器平滑迁移到独立 worker 进程或外部队列。

## 九、注意事项

- `tenant_access_token` 在服务内存中缓存，并在过期前自动刷新。
- 每个飞书事件都有唯一 `event_id`，需要去重避免重复分析。
- 当前正式链路仅消费数据库中保存的飞书应用配置、Webhook Token、Base 配置与 OAuth 授权；服务器全局 `.env` 不再承载任何用户级飞书密钥。
- OAuth 默认 scope 已包含 `offline_access`，用于换取 `refresh_token` 并自动刷新 `user_access_token`。
- 多租户阶段下，所有用户级敏感字段都必须加密后写库；前端仅展示脱敏值，日志不得打印完整 token 或 secret。
- Webhook 路由入口现已统一基于 `Webhook Verification Token -> integration` 动态识别集成；OAuth 回调统一要求 `state -> user_id + integration_id` 绑定当前集成。
- 当 webhook 命中数据库中的某个集成后，系统会回写该集成的 `lastWebhookReceivedAt`，并把 `webhookStatus / eventSubscriptionStatus` 更新为 `success`，便于在 `/feishu-config` 中直接观察回调是否打通。
- Base 初始化当前会自动补齐以下字段：`会议ID`、`会议主题`、`开始时间`、`结束时间`、`组织者`、`处理状态`、`会议文字稿`、`分析摘要`、`报告链接`、`JSON数据`、`错误信息`。
- 平台数据库需要保留会议主索引；飞书 Base 负责给用户展示结果和承接写回。
- 当前实现通过 `meeting_id -> recording.url -> minute_token -> transcript` 获取文字稿，不再依赖标题、组织者和时间窗搜索妙记。
- 若未开通 `vc:record:readonly` 或 `minutes:minutes.transcript:export`，系统会将错误写回 `错误信息` 字段，提示检查应用权限或资源范围。
- 若重试耗尽或分析最终失败，需要将失败状态和 `错误信息` 写回多维表格，方便排障。
- 结构化监控日志目前直接输出到应用日志，建议部署时重点关注以下事件：`meeting_pipeline_task_duplicate_ignored`、`recording_fetch_finished`、`transcript_export_finished`、`analysis_failed`、`meeting_pipeline_failed`、`startup_recovery_scan_finished`。
- 当前版本已将会议主链路的任务状态、重试时间、Base 记录 ID、minute token 和错误信息写入数据库，并由 `meetingPipelineWorker.ts` 统一轮询执行；后续若进入稳定生产阶段，建议继续把执行器单独部署为独立 worker / 队列服务。
