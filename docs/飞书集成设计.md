# 飞书集成设计文档

> 最后更新时间：2026-06-18

## 一、概述

当前正式飞书集成方案为 `Webhook + OpenAPI + 服务器环境变量`。

目标闭环：

- 接收会议结束事件
- 将会议基础信息写入多维表格
- 搜索与该会议匹配的妙记
- 导出妙记文字稿
- 调用豆包分析
- 回写多维表格
- 打开报告页

## 二、技术架构

### 2.1 当前正式方案

当前生产链路统一使用 Webhook 接收 `vc.meeting.participant_meeting_ended_v1`，再由服务端通过 OpenAPI 搜索妙记、导出文字稿，并写入多维表格。

```text
飞书事件
  ↓
/api/feishu/webhook
  ↓
webhookProcessor.ts
  ↓
minutes search / transcript / analysisService.ts / bitableOpenApi.ts
  ↓
多维表格 + 报告页
```

### 2.2 为什么使用会议结束事件

当前自动化链路以“用户参与的会议已经结束”为起点。事件体中直接提供 `meeting.id`、`topic`、`start_time`、`end_time`、`owner.id.open_id`、`host_user.id.open_id`，足以完成建档与后续妙记搜索准备。

## 六、多维表格结构

当前运行时依赖一张会议信息表，字段如下：

| 字段名 | 字段类型 | 用途 |
|-------|---------|------|
| 会议ID | 文本 | 飞书会议唯一标识 |
| 会议主题 | 文本 | 会议标题 |
| 开始时间 | 日期时间 | 会议开始时间 |
| 结束时间 | 日期时间 | 会议结束时间 |
| 组织者 | 文本 | 会议拥有者或主持人的 open_id |
| 处理状态 | 单选 | 会议结束 / 获取文字稿中 / 分析中 / 已完成 / 失败 |
| 会议文字稿 | 文本 | 妙记转写全文 |
| 分析摘要 | 文本 | LLM 分析摘要 |
| 报告链接 | URL/文本 | `/report?recordId=xxx` 报告页面链接 |
| JSON数据 | 文本 | 完整 AnalysisResult JSON |
| 错误信息 | 文本 | 异常原因，用于排障 |

## 七、完整流程

### 7.1 初始化流程

1. 在飞书开放平台创建企业自建应用。
2. 配置权限与事件订阅。
3. 配置 Webhook 回调地址与 Verification Token。
4. 手工准备 Base，并将 `FEISHU_BASE_APP_TOKEN` / `FEISHU_MEETING_TABLE_ID` 写入服务端环境变量。
5. 若要启用 `user_access_token` 自动续期，需要额外准备 `FEISHU_USER_REFRESH_TOKEN`，并在飞书授权时开通 `offline_access`。

### 7.2 会议分析流程

1. 飞书发送 `vc.meeting.participant_meeting_ended_v1`。
2. `/api/feishu/webhook` 验签、去重、入队。
3. 服务端从事件体中直接提取 `meeting.id`、`topic`、`start_time`、`end_time`、`owner.id.open_id`；若 `owner.id.open_id` 缺失，则回退到 `host_user.id.open_id`，再回退到 `operator.id.open_id`。
4. 服务端先把 `会议ID`、`会议主题`、`开始时间`、`结束时间`、`组织者` 写入多维表格，并将状态写为 `会议结束`。
5. 首次建档时以 `meeting.id` 作为幂等键防止重复创建；一旦 Base 返回 `recordId`，后续状态更新、恢复链路和继续处理都优先使用 `recordId` 命中同一条记录。
6. 服务端在会议结束后第 `2`、`3`、`4`、`5` 分钟各执行一次妙记搜索。搜索请求使用：
   - `query = 会议主题`
   - `filter.owner_ids = [组织者 open_id]`
   - `filter.create_time.start_time = 会议结束时间`
   - `filter.create_time.end_time = 当前搜索时刻`
7. 如果搜索命中结果，则取第一条结果中的 `token` 作为 `minute_token`。
8. 服务端调用 `minutes/v1/minutes/:minute_token/transcript?need_speaker=true&need_timestamp=true&file_format=txt` 导出文字稿，状态变更为 `获取文字稿中`。
9. 调用豆包 API 分析，状态变更为 `分析中`；若豆包返回临时性错误，会在服务内做最多 3 次短间隔重试。
10. 回写 `会议文字稿`、`分析摘要`、`报告链接`、`JSON数据`，状态变更为 `已完成`。
11. 用户通过 `/report?recordId=...` 打开报告页。

### 7.3 可靠性机制

1. `搜索妙记` 接口要求使用 `user_access_token`；若提供了 `FEISHU_USER_REFRESH_TOKEN`，服务端会在令牌过期前自动刷新，并在 401/403 时强制刷新后重试一次。
2. `导出妙记文字记录` 接口支持 `user_access_token` 或 `tenant_access_token`；服务端优先使用用户身份请求，失败后再回退到应用身份。
3. 若第 2/3/4/5 分钟窗口内仍未搜到妙记，系统不会提前写成 `失败`，而是继续等待下一次搜索窗口；若 5 分钟窗口结束仍无结果，则写入业务可理解的失败原因。
4. 若导出文字稿时飞书返回 `minute not ready` 或临时性 5xx/429 错误，系统会继续等待下一次搜索窗口；若所有窗口耗尽仍未就绪，则写回失败状态。
5. 单进程内对同一会议持有处理锁，避免同一会议被并发导出、并发分析；锁超时后允许后续事件重新进入链路，防止记录长期卡住。
6. Base 回读时会统一把文本单元格和富文本片段数组归一化为纯字符串，避免启动恢复时因字段返回格式差异读不出 `meetingId`、`会议主题` 或 `组织者`。
7. Webhook 处理链路输出结构化监控日志，统一记录 `meetingId`、`recordId`、`attempt`、`organizerSource`、`minuteToken`、错误类型和重试行为，便于在应用日志中追踪搜索、导出、分析和恢复过程。
8. 服务启动时会扫描 Base 中状态为 `会议结束 / 获取文字稿中 / 分析中` 的记录，并自动恢复未完成链路：
   - 若记录处于 `分析中` 且已写入 `会议文字稿`，则直接从文字稿继续分析并回写结果；
   - 其他活跃状态会保留已知 `recordId` 直接继续“搜索妙记 / 导出文字稿 / 分析”主流程，不再优先回退到 `meetingId -> 搜索记录` 的旧路径。
9. 启动恢复默认开启，可通过环境变量 `FEISHU_ENABLE_STARTUP_RECOVERY=false` 显式关闭；该恢复机制依赖当前 Base 中已有的 `处理状态`、`会议文字稿`、`错误信息` 等字段，不要求新增多维表格字段。

## 九、注意事项

- `tenant_access_token` 在服务内存中缓存，并在过期前自动刷新。
- 每个飞书事件都有唯一 `event_id`，需要去重避免重复分析。
- `搜索妙记` 依赖 `user_access_token`；推荐同时提供 `FEISHU_USER_REFRESH_TOKEN`，以便服务在运行时自动续期访问令牌。
- 当前文档列出的妙记搜索接口不支持直接按 `meeting_id` 过滤，因此运行时采用“会议标题 + 组织者 open_id + 创建时间窗”的组合搜索。
- 当前实现取搜索结果的第一条 `token` 作为后续导出目标；若未来接口新增更强的 `meeting_id` 关联字段，应优先切换为强匹配。
- 若妙记未共享给当前用户，或未授予导出文字稿权限，系统会将错误写回 `错误信息` 字段，提示联系组织者共享妙记并授予导出权限。
- 若重试耗尽或分析最终失败，需要将失败状态和 `错误信息` 写回多维表格，方便排障。
- 结构化监控日志目前直接输出到应用日志，建议部署时重点关注以下事件：`minutes_search_finished`、`transcript_export_finished`、`analysis_failed`、`meeting_pipeline_failed`、`startup_recovery_scan_finished`。
- 启动恢复面向“单机 + 单用户”第一阶段，主要解决服务重启后内存定时器丢失的问题；事件去重集合和搜索重试计划仍然是进程内状态，后续若进入稳定生产阶段，建议迁移到队列或数据库。
