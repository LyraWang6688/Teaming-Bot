# 飞书集成设计文档

> 最后更新时间：2026-06-17

## 一、概述

本文档描述组队会议动力分析系统与飞书的集成方案，包括：
- Webhook 事件接收机制
- 前端配置页设计
- 多维表格结构设计
- 完整的数据流程

---

## 二、技术架构

### 2.1 长连接模式 vs Webhook 模式

| 对比项 | Webhook 模式 | 长连接模式 |
|-------|-------------|-----------|
| 公网域名 | 需要 | **不需要** |
| 回调地址 | 需要配置 | **不需要** |
| 本地开发 | 无法使用 | **可以直接使用** |
| 配置复杂度 | 高 | **低** |

**当前正式方案：Webhook 模式**。长连接/CLI 模式已经标记为废弃，仅保留作历史参考，不再作为前端配置页入口，也不再建议继续扩展。

原因是 `lark-cli event consume` 无法消费 `vc.bot.meeting_ended_v1`，因此不能覆盖“用户是会邀人但未实际参会”的会议结束场景；而当前业务必须覆盖该场景，所以生产链路统一收口到 `Webhook + OpenAPI`。

### 2.2 事件驱动架构

```
┌─────────────────────────────────────────────────────────────────┐
│                    事件驱动流程                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   用户开会中... → 会议结束 → 事件触发 → 后端处理                   │
│                           ↓                                     │
│              vc.meeting.participant_meeting_ended_v1           │
│                           ↓                                     │
│                    记录会议信息                                   │
│                           ↓                                     │
│                    飞书生成转录稿                                 │
│                           ↓                                     │
│                    纪要生成完成                                   │
│                           ↓                                     │
│                    vc.note.generated_v1                         │
│                           ↓                                     │
│              获取转录稿 → AI Provider 分析 → 写入多维表格          │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 三、订阅事件清单

### 3.1 必须订阅的事件

| 事件名称 | 事件 ID | 触发时机 | 用途 |
|---------|--------|---------|------|
| 参与的会议结束 | `vc.meeting.participant_meeting_ended_v1` | 用户参与的会议结束时 | 获取会议基本信息 |
| 会议结束 | `vc.bot.meeting_ended_v1` | 会议整体结束时 | 覆盖被邀请但未实际参会场景 |
| 纪要生成 | `vc.note.generated_v1` | 智能纪要/转录稿生成完成时 | 获取转录稿 token 并触发分析 |

> 说明：当前 `lark-cli event consume` 仅支持 `vc.meeting.participant_meeting_ended_v1` 和 `vc.note.generated_v1`，不支持消费 `vc.bot.meeting_ended_v1`。因此生产链路采用 Webhook 接收三个事件。

### 3.2 事件数据结构

#### vc.meeting.participant_meeting_ended_v1

```json
{
  "event_id": "xxx",
  "meeting_id": "123456789",
  "meeting_no": "147 738 973",
  "topic": "会议主题",
  "start_time": "2026-06-07T15:00:00+08:00",
  "end_time": "2026-06-07T16:00:00+08:00",
  "calendar_event_id": "xxx",
  "timestamp": "1749273600000",
  "type": "vc.meeting.participant_meeting_ended_v1"
}
```

#### vc.note.generated_v1

```json
{
  "event_id": "xxx",
  "note_id": "xxx",
  "note_token": "doccnxxx",
  "verbatim_token": "doccnxxx",
  "note_source": {
    "source_type": "xxx",
    "source_entity_id": "xxx"
  },
  "timestamp": "1749274200000",
  "type": "vc.note.generated_v1"
}
```

## 四、所需权限清单

### 4.1 多维表格权限

| 权限 ID | 权限名称 | 用途 |
|--------|---------|------|
| `bitable:app:write` | 创建多维表格 | 初始化时创建多维表格 |
| `bitable:app:read` | 读取多维表格 | 读取多维表格信息 |
| `bitable:table:write` | 创建数据表 | 创建会议信息表和授权信息表 |
| `bitable:table:read` | 读取数据表 | 读取数据表结构 |
| `bitable:field:write` | 创建字段 | 创建数据表字段 |
| `bitable:record:write` | 写入记录 | 写入会议信息和分析结果 |
| `bitable:record:read` | 读取记录 | 读取用户授权信息 |

### 4.2 会议权限

| 权限 ID | 权限名称 | 用途 |
|--------|---------|------|
| `vc:meeting:readonly` | 读取会议信息 | 获取会议详情 |
| `vc:meeting.meetingevent:read` | 会议事件读取 | 订阅会议事件 |
| `vc:note:read` | 读取会议纪要 | 获取转录稿 token |

### 4.3 用户信息权限

| 权限 ID | 权限名称 | 用途 |
|--------|---------|------|
| `contact:user.base:readonly` | 读取用户基本信息 | 获取用户姓名等信息 |

### 4.4 文档权限

| 权限 ID | 权限名称 | 用途 |
|--------|---------|------|
| `docs:doc:readonly` | 读取文档内容 | 读取相关文档内容 |
| 妙记转录稿导出权限 | 导出妙记文字记录 | 获取逐字转录稿文本 |

## 六、多维表格结构设计

| 字段名 | 字段类型 | 字段说明 | 存储内容 |

| 组织者 | 文本 | organizer | 会议主持人姓名 |

| 分析摘要 | 多行文本 | analysis_summary | LLM 分析摘要 |
| 报告链接 | 文本 | report_url | `/report?recordId=xxx` 报告页面链接 |

## 八、飞书 Webhook 与 CLI 调试参考

### 8.1 Webhook 接口

生产 Webhook 地址：

```text
https://meeting.bamamei.online/api/feishu/webhook
```

Webhook 接口职责：

1. URL 验证与 token 校验
2. event_id 去重
3. 创建异步处理任务
4. 立即返回 200

耗时步骤（拉取转录稿、LLM 分析、写入多维表格）由后台任务处理，避免 Webhook 超时。

当前私有服务器部署采用 Docker Compose，Nginx 统一接入公网 80/443 后反向代理：

```text
Nginx 443
  ↓
127.0.0.1:3011
  ↓
Docker 容器内 Next.js :3000
```

### 8.2 生产环境变量

```env
PROJECT_PUBLIC_URL=https://meeting.bamamei.online

ANALYSIS_PROVIDER=doubao
DOUBAO_API_KEY=
DOUBAO_BASE_URL=https://ark.cn-beijing.volces.com/api/v3
DOUBAO_MODEL=doubao-seed-1-8-251228

FEISHU_APP_ID=
FEISHU_APP_SECRET=
FEISHU_WEBHOOK_VERIFICATION_TOKEN=
FEISHU_BASE_APP_TOKEN=
```

当前正式分析链路只保留火山引擎豆包 OpenAI-compatible API。运行时配置统一由服务器环境变量注入，不再依赖前端页面触发 CLI 初始化。

## 九、注意事项

### 9.1 Token 管理

- 生产 Webhook 链路使用 `FEISHU_APP_ID` / `FEISHU_APP_SECRET` 换取 `tenant_access_token`
- `tenant_access_token` 在服务内存中缓存，并在过期前自动刷新
- 如果后续恢复多租户用户授权模式，才需要继续维护用户 access_token / refresh_token

### 9.2 事件去重

- 每个事件都有唯一的 event_id
- 需要去重处理，避免重复分析

### 9.3 错误处理

- 会议可能没有生成转录稿（用户未开启录制或没有妙记权限）
- 转录稿生成可能延迟较长时间
- 分析输入只能使用逐字转录稿，不能使用智能纪要正文 fallback
- 需要记录处理状态，方便排查问题
