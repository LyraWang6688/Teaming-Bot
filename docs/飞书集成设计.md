# 飞书集成设计文档

> 最后更新时间：2026-07-24

## 一、方案与边界

飞书集成统一使用官方 `@larksuiteoapi/node-sdk` 1.71.0：

- `registerApp` 提供一键创建应用，业务用户不需要进入开放平台逐项配置。
- 标准 OAuth 授权码流程取得真实 `access_token`、`refresh_token` 与有效期。
- `TokenService` 从数据库读取和刷新用户令牌；容器文件系统不再是凭证事实来源。
- `Client.request + withUserAccessToken` 调用妙记与 Base OpenAPI。
- `client.im.message.create` 以应用机器人身份向授权用户主动发送会议报告通知卡片。
- `POST /open-apis/minutes/v1/minutes/subscription` 订阅当前授权用户的 `minutes.minute.generated_v1`。
- `WSClient + EventDispatcher` 消费 `minutes.minute.generated_v1`。

平台域名、数据库连接和加密主密钥属于全局配置；App ID/App Secret、OAuth 令牌、组织目标和检查状态均按 `integrationId` 落库。App Secret 与令牌统一使用 `encrypt()` / `decrypt()`，前端只接收脱敏值。

## 二、六段顺序状态机

初始化严格按以下顺序执行，不并行启动事件消费：

1. **创建应用**：`registerApp` 返回 App ID/App Secret 后立即加密落库；后端自动配置 OAuth 回调、Refresh Token 开关、WebSocket 订阅方式并提交发布。
2. **用户授权**：生成带一次性 `state` 的授权链接；回调换取真实令牌、读取授权用户身份并加密落库。
3. **选择组织**：用户选择服务器项目配置中已有的组织目标，数据库保存 `selectedOrgTargetId`。
4. **Base 可访问校验**：每个用户都使用自己的用户令牌读取目标 Base 和 Table 字段列表，请求成功即表示“当前用户 + 当前应用”可以访问该目标表。系统不判断该用户是否为项目首个用户，也不比较字段名称、类型或模板完整性。
5. **妙记事件订阅**：使用当前用户令牌调用 `POST /open-apis/minutes/v1/minutes/subscription`，显式订阅 `minutes.minute.generated_v1`。
6. **事件长连接**：只有前五项通过后才创建 `WSClient`；`onReady` 后将 `eventSubscriptionStatus` 写为 `success`。

后端为第 5、6 段分别持久化 `minuteSubscriptionStatus` 与 `eventSubscriptionStatus`。前者表示飞书服务端是否已为当前授权用户建立妙记事件订阅关系，后者只表示本系统的 SDK WebSocket 消费连接是否就绪；两者不得合并成同一个数据库状态。

对于同一飞书人反复创建新应用的场景，系统额外维护 `feishu_integrations.is_active / activated_at / superseded_at / superseded_by_integration_id`。按 `feishu_union_id + selected_org_target_id` 分组后，只保留最新创建的一条集成为当前生效版本；同组旧集成会被标记为 superseded，不再继续消费会议事件。

前端烟花只由最终 `eventSubscriptionStatus === success` 的状态跃迁触发。前五项是后端建立长连接的必要门禁，因此前端不再重复组合一组并行布尔值判断“初始化完成”。

操作顺序不仅由前端按钮状态控制，后端接口也执行相同门禁：应用校验未通过不能发起授权，用户授权未完成不能选择组织，Base/权限未通过不能订阅妙记事件，前五项未全部通过不能启动事件长连接。

组织目标配置只保存项目、组织、Base App Token、Table ID、Base URL 和启用状态。Base 可访问结果属于用户集成级事实，只写入 `feishu_integration_checks.base_status`；同一组织中的不同用户必须分别使用自己的 OAuth Token 检查，不能共享或缓存另一个用户的成功结果。字段列表接口仅作为目标 Table 可读性探测，不承担字段模板校验。

## 三、应用创建与自动配置

`registerApp` 使用：

- `createOnly: true`，避免误绑定已有应用。
- `addons.preset: false`，从最小基座开始声明所需权限和事件。
- `addons.scopes.user` 预填用户权限。
- `addons.scopes.tenant` 预填应用自身配置与机器人发消息所需的应用权限。
- `addons.events.items.user` 预填妙记生成事件。
- `application.v7.applicationConfig.patch` 配置 OAuth 回调、允许刷新用户令牌及 WebSocket 订阅方式。
- `application.v7.applicationPublish.create` 提交上述敏感配置使其生效；由于创建模板包含机器人能力，发布请求必须同时声明移动端与 PC 端默认能力为 `bot`。

SDK 的 Device Grant 轮询由 SDK 自己维护，遵循服务端 `interval/slow_down`；项目不再用 12 秒超时反复杀死授权进程。SDK 返回凭据后会立即触发服务端最终化并写数据库，前端轮询仅用于展示状态。

应用自身配置使用的应用身份权限与最终用户授权分层：

| 类型 | 权限 ID | 用途 |
| --- | --- | --- |
| 应用身份 | `application:application:self_manage` | 管理并发布当前应用自身配置 |
| 应用身份 | `application:application:patch` | 调用 `applicationConfig.patch` 自动写入 OAuth 回调、Refresh Token 开关和 WebSocket 订阅方式 |
| 应用身份 | `im:message:send_as_bot` | 以当前应用机器人身份向授权用户主动发送会议报告通知卡片 |

`registerApp.addons` 只能增量预填公开权限、事件和回调，不能携带 OAuth 重定向地址、订阅方式等敏感配置。因此 `application:application:patch` 是保持“一键创建并自动配置”交互所需的初始化权限，不属于最终用户 OAuth 权限，也不用于读取妙记或 Base 数据。

若应用最终配置失败，注册任务进入 `failed` 终态，后台完成回调不得把它重置为 `completed`，前端停止轮询并展示失败提示，避免下一次轮询重复调用相同的配置或发布接口。运行日志按 `application_config_patch` 与 `application_publish` 区分操作阶段，并记录飞书返回的非敏感字段校验信息。

应用配置与发布只允许在注册最终化阶段执行一次。后续系统真实检查只通过只读接口验证应用凭证、OAuth、Base、权限和事件监听状态，禁止在检查过程中再次调用应用配置或发布接口。前端使用同步请求锁避免同一页面重复发起检查，后端再按 `userId + integrationId` 合并同一进程内的并发检查；该单飞仅用于并发协调，最终检查状态仍以数据库记录为唯一事实来源。

## 四、用户权限

当前只讨论并使用下列用户身份权限：

| 权限 ID | 用途 |
| --- | --- |
| `auth:user.id:read` | OAuth 完成后读取授权用户 open_id 与基础身份 |
| `minutes:minutes.basic:read` | 读取妙记基础信息并满足妙记事件访问要求 |
| `minutes:minutes.transcript:export` | 导出妙记转写文字 |
| `bitable:app` | 读取和写入用户有权访问的多维表格 |
| `offline_access` | 获取 Refresh Token，支持长期续期 |

权限清单由 `integrationConstants.ts` 单点定义，应用创建 addons、OAuth scope、数据库记录和后端校验共同消费，禁止前后端分别硬编码不同清单。

会议报告通知不复用用户令牌，而是使用当前 `integrationId` 对应应用的机器人身份发出；接收者使用 OAuth 回调落库的 `authorized_open_id`，避免额外依赖 `user_id` 映射。

## 五、OAuth 与 TokenService

授权入口构造 `https://accounts.feishu.cn/open-apis/authen/v1/authorize`，回调固定为：

```text
${PROJECT_PUBLIC_URL}/api/feishu/oauth/callback
```

OAuth `state` 只在数据库保存哈希，10 分钟过期且仅可消费一次。回调完成以下动作：

1. 校验并消费 `state`。
2. 使用 SDK OAuth v3 token 接口交换授权码。
3. 使用 `authen.v1.userInfo.get` 读取授权主体。
4. 加密保存 Access Token、Refresh Token、scope 与两类过期时间。
5. 更新用户身份、授权检查和审计日志。
6. 重定向 `/feishu-config?integrationId=...&oauth=success`。

`TokenService` 在用户 API 调用前执行：

- Access Token 剩余不足 5 分钟时自动刷新。
- 保存轮换后的 Refresh Token 和新有效期。
- 单个 `integrationId` 的并发刷新使用进程内 single-flight；数据库仍是唯一事实来源。
- Refresh Token 缺失、过期或刷新失败时，将授权标记为 `reauthorization_required`。
- Refresh Token 缺失的授权会被识别为不可续期，要求用户重新授权一次。

## 六、用户级妙记事件订阅

`minutes.minute.generated_v1` 除了需要在应用配置中声明事件、并把订阅方式设置为 WebSocket 外，还必须按当前授权用户执行一次：

```text
POST /open-apis/minutes/v1/minutes/subscription
{
  "event_type": "minutes.minute.generated_v1"
}
```

该接口使用 `user_access_token` 调用，作用是把“当前授权用户相关的妙记资源变更”显式订阅给当前应用。若缺少这一步，即便 `WSClient` 已 ready，服务器侧也可能完全收不到妙记生成事件。

当前实现将这一步放入真实检查链路：应用凭证通过、OAuth 有效、已选择组织且目标 Base 可访问、OAuth scope 校验通过后，先调用订阅接口，再启动 `WSClient`。调用结果写入 `feishu_integration_checks.minute_subscription_status`：成功为 `success`，调用失败为 `failed`，尚未满足前置条件或授权/组织发生变化时为 `pending`。

数据库迁移会把历史上 `event_subscription_status = success`，或检查详情明确记录用户订阅成功的集成，回填为 `minute_subscription_status = success`，避免已正常运行的集成在升级后退回未初始化状态。

## 七、事件订阅与消费门禁

应用配置中存在 `minutes.minute.generated_v1` 只表示“允许订阅”，不表示进程已经消费事件。真正的消费由 `eventListenerManager.ts` 创建 `WSClient` 后开始。

启动前必须同时满足：

```text
appCredentialStatus = success
oauthStatus = authorized（且 TokenService 校验真实令牌可用）
selectedOrgTargetId 非空
baseStatus = success
permissionStatus = success
minuteSubscriptionStatus = success
```

服务启动恢复同样重新读取这些数据库门禁，不会因为数据库曾经存在 integration 就直接监听。除上述六段门禁外，事件监听还要求当前集成满足 `is_active = true`；服务启动恢复只会拉起 active integration，真实检查通过后也会先按 `feishu_union_id + selected_org_target_id` 重新计算“谁是最新集成”，再为该 active integration 启动监听，并主动停止同组旧 listener。连接状态通过 SDK 回调维护：

- `onReady/onReconnected`：事件状态成功，集成标记初始化完成。
- `onReconnecting`：事件状态暂时回到 pending。
- `onError`：保存内部错误类型和可操作提示。

运行时客户端 Map 只保存当前 WebSocket 对象，不保存授权事实。若未来横向扩容为多个应用实例，需要再增加数据库租约，保证同一 integration 只有一个活跃消费者。

## 八、多维表格与分析流程

组织到 Base 的映射仍由服务器项目 JSON 导入数据库，用户只选择组织，不创建个人表格。初始化门禁目前只验证目标 Base 可读取；事件到达后的写入链路仍使用 `bitable:app`。

系统只通过 `feishu_integrations.selected_org_target_id -> feishu_project_org_targets` 定位 Base。旧版按单个集成保存 Base Token/Table ID、自动创建个人 Base，以及缺少组织参数时回退到集成级 Base 的链路均已删除。报告链接必须携带生成任务绑定的 `orgTargetId`，以稳定读取当次会议写入的组织 Base。

自动分析链路：

```text
  minutes.minute.generated_v1
    -> 当前授权用户已订阅 minutes subscription
    -> active integration listener
    -> eventListenerManager
  -> meeting_pipeline_tasks（数据库幂等与重试）
  -> minute_token 导出文字稿
  -> analysisService
  -> Base 回写
  -> 应用机器人向授权用户发送报告通知卡片
  -> 报告页
```

Base 业务字段保持：会议ID、处理状态、会议文字稿、分析摘要、报告链接、JSON数据、错误信息。

会议报告通知在 `reportUrl` 已生成且 Base 写回成功后触发。通知文案第一版保持简单稳定：

- 标题：`小组会议动力分析报告 已出炉`
- 正文：`会议ID：xxx`
- 按钮：`点击查看报告`

通知发送失败只写运行日志与审计日志，不回滚 Base 写回结果，也不把主分析链路改判为失败。

## 九、前端交互

- 三个用户动作依次为创建应用、用户授权、选择组织；后两步由系统自动完成。
- 创建完成后通过局部状态和接口重新读取更新按钮，不执行 `window.location.reload()`。
- OAuth 完成后飞书回调直接重定向配置页；原配置页也可通过轻量状态轮询即时更新。
- 重新授权或更换组织时立即停止旧连接，将 Base、权限、妙记订阅和事件长连接状态重置为 pending，再按顺序重新校验。
- 同一飞书人、同一组织目标下，只有当前 active integration 具备监听资格；新应用接管后会主动停止同组旧 listener。
- 页面只消费内部枚举状态，不依赖飞书或 SDK 的原始错误字符串。

## 十、部署与审计

- Docker 镜像不再安装 `@larksuite/cli`，Compose 不再挂载 `.lark-cli` Volume。
- `feishu_integrations` 不再保存 CLI profile/config 目录或集成级 Base Token/Table ID；Base 凭证只存在于组织目标配置中。
- 容器重建后从数据库恢复应用凭证、用户令牌和初始化门禁。
- OAuth、Token 刷新、应用配置、检查、事件连接、Base 操作均写结构化日志或审计日志。
- 会议报告通知单独输出 `report_notification_started / succeeded / failed` 结构化日志，并写入 `meeting.report.notification.send` 审计动作。
- 日志禁止输出完整 App Secret、Access Token 或 Refresh Token。
- 事件监听排障统一输出 `reasonCode`、`blockedGate`、`prerequisiteFailures` 与当前检查状态：
  - `event_listener_gate_blocked` 表示前置门禁未过，长连接不会启动。
  - `event_listener_start_skipped` 表示服务启动恢复时跳过某个集成。
  - `event_listener_failed` / `event_listener_gate_failed` 表示门禁已过但 SDK WebSocket 建立或运行失败。
- 常见 `reasonCode` 包括 `organization_not_selected`、`oauth_refresh_token_missing`、`oauth_refresh_token_expired`、`base_access_failed`、`permission_scope_failed`。线上排查时先看 `blockedGate`，再看 `reasonCode` 与 `lastErrorMessage`。
- `APP_ENCRYPTION_KEY` 必须稳定保管；更换该密钥前需设计密钥轮换，否则历史密文不可解密。
